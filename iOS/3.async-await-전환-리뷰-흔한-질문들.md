기존 RxSwift·completion 콜백 기반 화면 하나를 async/await로 옮기고 나서 코드 리뷰를 받으면, 컴파일은 통과했는데도 "이거 원래 의도랑 다른 거 아니야?" 싶은 질문이 꼭 몇 개 나온다. 이 글은 그런 질문에 "이게 정답이다"라고 결론 내리기보다, **실제로 던져진 질문과 그때그때 나온 선택지**를 정리한 것이다. 각 질문마다 상황에 따라 답이 달라질 수 있으니, 체크리스트 삼아 읽으면 된다.

---

## 의문 1 — "원래는 두 호출이 동시에 진행됐는데, `await`로 묶었더니 순차 실행되는 거 아닌가?"

전환 전 코드가 이런 식이었다고 하자.

```swift
// Before — 두 호출이 서로 기다리지 않고 거의 동시에 진행됨
func viewWillAppear() {
    NotificationManager.shared.refreshSettingInfo()  // fire-and-forget
    refresh()                                         // 자체 API 호출 시작
}
```

이걸 기계적으로 async/await로 바꾸면 이렇게 되기 쉽다.

```swift
// After — 의도치 않게 직렬화됨
Task {
    await NotificationManager.shared.refreshSettingInfoAsync()
    await refresh()
}
```

겉보기엔 자연스러운 변환이지만, 두 작업이 서로 다른 상태(예: 기기 알림 권한 상태 vs 서버 설정값)를 갱신할 뿐 서로 결과를 참조하지 않는다면, `await`로 묶는 순간 **원래 없던 순서 의존성**이 생긴다. 두 번째 작업은 첫 번째 작업이 끝날 때까지 시작조차 안 하므로, 첫 화면 노출까지 걸리는 시간이 늘어난다.

**선택지**
- 서로 독립적이면 `Task`를 두 개로 분리해서 각자 병렬로 돌린다.
- 개수가 정해져 있고 최종 결과를 합쳐야 한다면 `async let`으로 병렬성을 유지한 채 한 줄로 기다린다.
- 정말로 순서가 필요하다면(둘째가 첫째 결과를 쓴다면) 지금처럼 이어서 `await`하는 게 맞다 — 다만 "정말 순서가 필요한지"를 한 번은 짚고 넘어가야 한다.

## 의문 2 — "async 함수를 동기 클로저(버튼 탭 핸들러 등)에서 부르려면 꼭 `Task{}`로 감싸야 하나?"

RxSwift의 `subscribe(onNext:)`나 UIKit 델리게이트 콜백은 전부 동기(sync) 클로저다. 그 안에서 async 함수를 `await` 없이 그냥 호출하면 컴파일 에러가 난다.

```swift
button.rx.tap
    .subscribe(onNext: { [weak self] in
        self?.handleTap()          // ❌ 컴파일 에러 — async 함수를 await 없이 호출
    })
```

`Task { }`가 이 둘 사이를 잇는 사실상 유일한 다리다. 다만 여기서 흔한 오해가 하나 있다: **`Task { await ... }`가 "끝날 때까지 기다린다"는 뜻이 아니라는 점**이다. `Task`는 작업을 새로 스케줄링하고 그 자리에서 즉시 반환한다. 즉 `subscribe(onNext:)` 클로저 자체는 블로킹 없이 바로 끝나므로, 이미 "기다리지 않는(fire-and-forget)" 동작 그대로다.

```swift
button.rx.tap
    .subscribe(onNext: { [weak self] in
        Task { await self?.handleTap() }   // ✅ 클로저는 즉시 반환, handleTap()은 백그라운드에서 진행
    })
```

**선택지**
- 위처럼 호출부에서 `Task{}`로 감싸는 게 가장 직관적이다.
- 호출부 보일러플레이트가 신경 쓰이면 (의문 5 참고) 오버로드나 RxCocoa `Binder` 같은 방식으로 감싸는 지점을 한 군데로 모을 수 있다.

## 의문 3 — "함수를 `@MainActor`로 바꿨는데, 실제 네트워크 처리가 백그라운드에서 되는 건 어디서 보장하나?"

```swift
@MainActor
func requestSaveContents() async {
    do {
        let result = try await apiCall()   // 이 네트워크 I/O는 어느 스레드?
        updateUI(result)
    } catch {
        showError()
    }
}
```

`@MainActor`를 붙이면 "이 함수 안의 코드는 메인에서 돈다"는 보장이 생기는데, 그러다 보니 "그럼 네트워크 요청도 메인에서 도는 거 아니야? 성능 괜찮나?"라는 의문이 자연스럽게 따라온다.

답은: **`@MainActor`는 백그라운드 실행을 막지 않는다.** 실제 네트워크 I/O는 URLSession(또는 그 위에 얹힌 Alamofire/Moya 같은 네트워킹 레이어)이 자체적으로 관리하는 백그라운드 큐에서 처리된다. 이건 Swift Concurrency와 무관하게 원래부터 그래왔던 부분이다. `@MainActor` 함수 안에서 그 요청을 `await`하면, 요청이 진행되는 동안은 실제로 백그라운드에서 돌고, **응답이 와서 함수가 재개(resume)되는 시점에** 런타임이 자동으로 메인 액터로 다시 홉(hop)시켜준다. 즉 "백그라운드에서 일하고, 결과를 받아 UI를 만질 때만 메인으로 돌아온다"는 게 정확히 분리되어 있다.

**선택지**
- 이 분리를 신뢰하고 그대로 쓰면 된다 — 별도로 `DispatchQueue.global()`에 태울 필요 없음.
- 다만 completion → continuation 브릿지를 손수 짤 때는 정말로 그 completion이 백그라운드에서 불리는 라이브러리인지 확인은 해두는 게 좋다(대부분의 네트워킹 라이브러리는 그렇다).

## 의문 4 — "async 메서드가 늘어날수록 `Task{}`도 똑같이 늘어나는데, 차라리 메서드는 sync로 두고 내부에서만 `Task`로 감싸면 안 되나?"

의문 2의 자연스러운 후속 질문이다. 매번 호출부에서 `Task { await self?.xxx() }`를 써야 한다면, 그냥 메서드 시그니처는 sync로 유지하고 **메서드 내부**에서 필요한 부분만 `Task{}`로 감싸면 되지 않느냐는 것.

```swift
// 대안: 메서드 자체는 sync, 내부에 Task 숨기기
func handleTap() {
    Task { await doNetworkWork() }
}
```

이건 실제로 유효한 선택지지만, **결과나 후속 처리가 필요 없는 진짜 fire-and-forget 메서드에 한해서만** 안전하다. 예를 들어 이런 메서드는 문제가 된다.

```swift
@MainActor
func requestSaveContents() async {
    do {
        let result = try await apiCall()
        updateUI(result)      // 성공 후 UI 갱신
        showToast()           // 성공 후 안내
    } catch {
        showError()           // 실패 후 UI 갱신
    }
}
```

이 메서드를 sync로 바꾸고 내부에 `Task { }`를 숨기면, 함수 자체는 네트워크 요청이 끝나기 **전에** 바로 리턴돼버린다. `updateUI`/`showToast`/`showError` 같은 후속 로직을 이 함수 안에 둘 방법이 없어지고, 결과적으로 다시 completion 클로저를 노출해야 한다 — 애초에 없애려던 콜백 중첩으로 되돌아가는 셈이다.

**선택지**
- **후속 처리(성공/실패에 따른 UI 갱신, 로깅, 에러 처리)가 있는 함수**는 시그니처 자체를 async로 열어야 한다.
- **정말 결과를 아무도 안 쓰는 순수 fire-and-forget**(예: 실패해도 무시하고 다음 기회에 재시도하는 로깅성 API 호출)이라면 sync + 내부 Task 방식도 합리적인 선택이다.
- 애매하면 "이 함수가 끝난 뒤에 뭔가 더 해야 하는가?"를 기준으로 판단하면 된다.

## 의문 5 — "그럼 결과를 기다리는 곳이 정말 없는 순수 이벤트 핸들러라면, 굳이 async로 열어둘 필요가 있나?"

의문 4에 대한 반론으로 나올 수 있는 질문이다. 예를 들어 버튼 탭 → 서버에 설정 반영까지 하는 메서드는 async가 맞지만, "그 결과를 또 기다리는 다른 메서드가 하나도 없는 진짜 터미널 핸들러"라면 굳이 async 시그니처를 유지할 필요가 없지 않냐는 것. 맞는 지적이다 — "누군가 기다려야 하니까 async"라는 논리는 이 경우엔 성립하지 않는다.

다만 별개로 async를 유지하는 편이 나은 이유가 있다.

1. **시그니처의 정직성**: sync 시그니처인데 내부에서 `Task{}`로 비동기 작업을 던지면, 함수만 봐서는 "호출하면 그 자리에서 다 끝난다"고 보이지만 실제로는 리턴 이후에도 작업이 계속된다. 타입에 async가 드러나 있으면 이런 오해가 안 생긴다.
2. **테스트 용이성**: async로 남겨두면 테스트 코드에서 `await`로 직접 호출해서 그 다음 줄에 바로 상태를 검증할 수 있다. sync + 내부 Task 방식은 리턴 시점에 아직 내부 작업이 안 끝나 있어서, 테스트 쪽에서 폴링이나 expectation이 필요해진다.

다만 이 둘은 "반드시 그래야 하는" 이유라기보단 스타일에 가깝다. 팀 컨벤션으로 "진짜 터미널 핸들러는 sync + 내부 Task"를 선호한다면 그 방향도 타당하다.

## 의문 6 — "호출부마다 반복되는 `Task { await self?.xxx() }` 보일러플레이트를 줄일 방법은 없나?"

몇 가지 방식이 논의됐다. 결론부터 말하면 **런타임에 async 여부를 검사하는 방법은 없고, 전부 컴파일 타임 오버로드 분기로 처리**해야 한다.

**방법 A — sync/async 오버로드를 직접 만들기**

```swift
extension ObservableType where Element == Void {
    func bind(with owner: AnyObject, onNext: @escaping () -> Void) -> Disposable { ... }
    func bind(with owner: AnyObject, onNext: @escaping () async -> Void) -> Disposable {
        subscribe(onNext: { [weak owner] in
            guard owner != nil else { return }
            Task { await onNext() }
        })
    }
}
```

클로저 안에 `await`가 있으면 Swift가 자동으로 async 타입으로 추론해서 두 번째 오버로드가 선택된다 — 컴파일러가 알아서 갈라주는 것이지, 우리가 런타임에 분기하는 게 아니다.

**방법 B — 클로저 대신 메서드 레퍼런스 자체를 넘기기**

```swift
tap.bind(with: self, SomeViewController.handleTap)
```

이것도 가능하다. 다만 한 가지 함정이 있다: 인스턴스 메서드 안에서 `Self.handleTap`(대문자 Self)처럼 쓰면, 이건 unbound 레퍼런스로 안 만들어지고 암묵적으로 `self.handleTap`(이미 바인딩된 형태)으로 해석돼 컴파일 에러가 난다. 이 패턴을 쓰려면 실제 타입명(`SomeViewController.handleTap`)을 그대로 써야 한다.

**방법 C — RxCocoa가 이미 제공하는 `Binder` 타입 활용**

가장 적은 코드로 끝나는 방법이다. `Binder`는 애초에 "약한 참조로 타깃을 잡고, 기본적으로 메인 스케줄러에서 실행한다"는 걸 라이브러리 차원에서 이미 보장해준다(RxSwift 소스를 보면 `weak var weakTarget = target`으로 내부 구현돼 있다).

```swift
extension SomeViewController {
    var handleTapBinder: Binder<Void> {
        Binder(self) { owner, _ in
            Task { await owner.handleTap() }
        }
    }
}

// 호출부
button.rx.tap.bind(to: handleTapBinder).disposed(by: disposeBag)
```

`Binder`의 `binding` 클로저 자체는 여전히 sync라서 async 메서드를 부를 땐 결국 `Task{}`를 한 번은 써야 하지만, 그게 이 computed property 정의부 한 곳에만 있고 실제 tap 바인딩 호출부는 클로저·`weak self`·`Task` 없이 완전히 깨끗해진다.

**선택지**: 셋 다 동작한다. A/B는 새 공용 확장이 필요하고, C는 RxCocoa 기본 제공 타입만 쓰면 돼서 코드가 가장 적다. 프로젝트에 이미 유사한 확장이 있다면 그 컨벤션을 따르는 게 우선이다.

## 의문 7 (가장 중요한 사례) — "다 됐다 싶었는데 왜 화면 전환 도중에 크래시가 났지?"

전환 자체는 각 파일 단위로 보면 다 멀쩡해 보였는데, 실제로 실행해보니 이런 크래시가 났다.

```
Modifications to the layout engine must not be performed from a background thread
after it has been accessed from the main thread.
```

원인을 추적해보니 이랬다.

1. 어떤 매니저의 completion 기반 메서드를 async로 바꾸면서 `@MainActor`를 빼먹었다.
2. 그 메서드 안에서 `relay.accept(...)`를 호출하는데, 이 호출이 (네트워크 응답이 온) **백그라운드 스레드에서 그대로 실행**됐다.
3. 그 relay를 구독하고 있던 다른 화면의 코드가 **`.accept()`를 호출한 바로 그 스레드에서 동기적으로** 콜백을 실행했다.
4. 그 콜백 안에서 화면 전환(새 컨테이너 뷰컨트롤러 초기화 + `rootViewController` 교체)이 일어났는데, 이게 전부 백그라운드 스레드에서 벌어졌다.

여기서 흔히 하는 오해 하나: **"relay의 `accept()`는 알아서 메인 스레드로 보내주는 거 아니야?"** — 아니다. RxRelay 소스를 보면 `accept(_:)`는 그냥 내부 subject의 `onNext()`를 호출할 뿐이고, 스케줄러도 스레드 전환 로직도 전혀 없다. `.accept()`를 호출한 스레드가 곧 구독자 콜백이 실행되는 스레드다.

GCD/completion 시절엔 completion 클로저 안에 `DispatchQueue.main.async`가 관성적으로 들어가 있어서 이런 실수가 티가 잘 안 났다. async/await로 옮기면서 그 안전장치를 걷어내고 `@MainActor`로 대체하는 과정에서, **딱 한 군데 `@MainActor`를 빼먹는 순간 그 보장이 조용히 사라진다** — 컴파일은 되고, 대부분의 상황에서는 우연히 잘 동작하다가, 특정 타이밍에서만 크래시로 터진다.

**체크리스트**
- UI 상태(relay, `@Published`, 델리게이트 콜백 등)를 갱신하는 async 함수는 **전부** `@MainActor`가 붙어 있는지 파일 단위가 아니라 **호출 체인 전체**를 따라가며 확인한다.
- "이 relay를 구독하는 화면이 또 있는가"를 항상 의심한다 — 내가 고친 화면은 안전해도, 같은 relay를 구독하는 **다른** 화면에서 사고가 날 수 있다.
- `.accept()`/`.onNext()` 같은 호출이 스케줄러 개입 없이 호출 스레드 그대로 전파된다는 걸 전제로, 의심 가는 지점은 직접 라이브러리 소스를 열어 확인하는 게 제일 빠르다.

| 병렬이 순차로 바뀜 | `await` 두 개를 이으면 직렬화된다 | Task 분리 / `async let` / 의도적 순차 |
| Task{}가 꼭 필요한가 | sync 컨텍스트에서 async 호출은 Task가 유일한 다리 | Task{}는 기다림이 아니라 즉시 반환 |
| @MainActor와 백그라운드 처리 | @MainActor는 함수 코드만 메인 보장, I/O 스레드는 무관 | 그대로 신뢰, 라이브러리 스레딩에 위임 |
| sync+내부Task vs async 시그니처 | 후속 처리 있으면 async 사실상 강제 | 순수 fire-and-forget이면 sync+내부Task도 유효 |
| 진짜 터미널 핸들러도 async? | 강제 아님, 스타일 문제 | 시그니처 정직성/테스트 용이성 vs 팀 컨벤션 |
| 호출부 보일러플레이트 | 컴파일타임 오버로드로 분기 | 커스텀 오버로드 / 메서드 레퍼런스 / Binder |
| 화면 전환 중 크래시 | @MainActor 누락 → relay가 백그라운드에서 accept | 호출 체인 전체에서 @MainActor 누락 여부 점검 |

정답을 하나로 정해두고 시작하기보다, 지금 손대는 코드가 "후속 처리가 있는가", "다른 화면도 같은 상태를 구독하는가"를 먼저 물어보면 대부분의 선택은 자연스럽게 정해진다.
