> 작성일: 2026-07-09
> 주제: Swift Concurrency / GCD 마이그레이션 / async-await / Actor

오래된 iOS 프로젝트는 대부분 `DispatchQueue`, `DispatchGroup`, `completion handler` 로 짜여 있다. Swift 5.5부터 `async/await`, `Actor`, `Task` 가 표준이 됐지만, 돌아가고 있는 프로덕션 코드를 하루아침에 갈아엎을 수는 없다. 전면 전환은 컴파일 에러 수백 개, 타이밍이 미묘하게 달라진 버그, 그리고 리뷰어도 파악 못 할 거대한 diff를 만든다.

이 글은 **"어디서부터 손대야 안전한가"** 를 리스크 낮은 순서대로 정리한 실전 노하우다. 개념 설명보다 "이 순서로 옮기면 되돌리기 쉽고, 사고 나도 범위가 좁다"에 초점을 맞춘다.

---

## 0. 왜 순서가 중요한가

GCD와 Swift Concurrency는 겉으로는 "비동기 처리"라는 같은 문제를 풀지만, 내부 동작 모델이 다르다.

| | GCD | Swift Concurrency |
|---|---|---|
| 실행 단위 | Dispatch Queue (스레드 추상화) | Task (협조적 스케줄링, 스레드는 런타임이 재사용) |
| 메인 스레드 진입 | `DispatchQueue.main.async` — **항상 큐에 넣고 다음 런루프에서 실행** | `@MainActor` — **이미 메인 액터 컨텍스트면 즉시 동기 실행될 수 있음** |
| 취소 | `DispatchWorkItem.cancel()` (제한적, 잘 안 씀) | `Task.cancel()` — 협조적 취소, 코드가 직접 확인해야 함 |
| 상호배제 | serial queue | `Actor` (재진입 가능 — await 지점에서 인터리빙 발생) |
| 데이터 경쟁 검증 | 런타임에만 (Thread Sanitizer) | **컴파일 타임** (Sendable, strict concurrency checking) |

표에서 굵게 표시한 두 줄이 마이그레이션 사고의 8할을 차지한다. **"똑같이 비동기인데 타이밍이 달라진다"** 는 점과, **"actor는 락이 아니라서 await 지점에서 상태가 바뀔 수 있다"** 는 점. 이걸 모르고 기계적으로 치환하면 겉으로는 컴파일이 되는데 미묘한 순서/타이밍 버그가 생긴다.

그래서 **"컴파일은 되지만 의미가 달라지는 지점"을 먼저 피해가며, 리스크가 낮은 코드부터 옮기는 순서**가 필요하다.

## 1. 적용 순서 로드맵

낮은 리스크 → 높은 리스크 순으로 8단계를 나눴다. 각 단계는 이전 단계가 어느 정도 안정화된 뒤 진행하는 걸 권한다. 한 PR에 여러 단계를 섞지 않는 게 핵심이다.

```
0단계  도구 준비 (strict concurrency 체크 최소 모드로 켜기)
1단계  순수 함수 · 유틸리티에 async 붙이기        ← 가장 안전
2단계  네트워크 레이어 completion handler → async/await 브릿징
3단계  UI 갱신 코드: DispatchQueue.main.async → @MainActor
4단계  fire-and-forget 작업: DispatchQueue.global().async → Task
5단계  DispatchGroup 병렬 처리 → async let / TaskGroup
6단계  공유 mutable state 보호용 serial queue → Actor
7단계  Combine/콜백 체인 → AsyncSequence
8단계  Strict Concurrency Checking → Swift 6 language mode  ← 가장 리스크 큼
```

리스크가 낮다는 기준은 "실패했을 때 되돌리기 쉬운가"와 "잘못돼도 파급 범위가 한 함수/한 화면에 갇히는가" 두 가지다. 반대로 6~8단계는 여러 파일에 걸친 상태 공유 방식 자체를 바꾸는 작업이라 신중해야 한다.

---

## 2. 1단계 — 순수 함수 · 유틸리티부터

가장 먼저 손대야 할 코드는 **입출력만 있고 공유 상태를 만지지 않는 함수**다. 이미지 리사이즈, 파싱, 포맷 변환 같은 것들.

```swift
// Before
func resizeImage(_ image: UIImage, to size: CGSize, completion: @escaping (UIImage?) -> Void) {
    DispatchQueue.global(qos: .userInitiated).async {
        let resized = image.resized(to: size)
        DispatchQueue.main.async {
            completion(resized)
        }
    }
}

// After
func resizeImage(_ image: UIImage, to size: CGSize) async -> UIImage? {
    await Task.detached(priority: .userInitiated) {
        image.resized(to: size)
    }.value
}
```

이 단계는 **호출부가 아직 GCD여도 상관없다.** 호출하는 쪽에서 `Task { let result = await resizeImage(...) }` 로 감싸기만 하면 되므로, 함수 하나 단위로 옮기고 그때그때 배포해도 안전하다. 컴파일러가 async 함수의 시그니처 변경을 강제해주기 때문에 "빼먹은 completion 호출" 같은 GCD 시절 흔한 버그도 자연스럽게 줄어든다.

> **노하우**: 이 단계에서 `Task.detached` 를 쓸지 그냥 `Task`를 쓸지 고민하지 말고, 유틸리티 함수 내부에는 `Task.detached` 를 기본값으로 써라. 상위 컨텍스트(actor)를 상속하지 않아 "어디서 호출해도 동일하게 동작"하는 예측 가능한 함수가 된다.

## 3. 2단계 — 네트워크 레이어 브릿징

completion handler 기반 네트워크 코드는 `withCheckedThrowingContinuation` 으로 감싼다. 이 단계부터는 **"completion을 정확히 한 번만 호출한다"** 는 기존 계약이 얼마나 지켜지고 있었는지가 드러난다.

```swift
// Before
func fetchUser(id: String, completion: @escaping (Result<User, Error>) -> Void) {
    let task = URLSession.shared.dataTask(with: url) { data, _, error in
        if let error = error { completion(.failure(error)); return }
        guard let data = data else { completion(.failure(NetworkError.noData)); return }
        do {
            completion(.success(try JSONDecoder().decode(User.self, from: data)))
        } catch {
            completion(.failure(error))
        }
    }
    task.resume()
}

// After
func fetchUser(id: String) async throws -> User {
    try await withCheckedThrowingContinuation { continuation in
        let task = URLSession.shared.dataTask(with: url) { data, _, error in
            if let error = error { continuation.resume(throwing: error); return }
            guard let data = data else {
                continuation.resume(throwing: NetworkError.noData); return
            }
            do {
                continuation.resume(returning: try JSONDecoder().decode(User.self, from: data))
            } catch {
                continuation.resume(throwing: error)
            }
        }
        task.resume()
    }
}
```

> **함정**: `continuation.resume()` 은 **반드시 정확히 한 번만** 호출돼야 한다. 두 번 호출하면 런타임이 크래시(디버그 빌드) 또는 未정의 동작을 일으킨다. `withCheckedThrowingContinuation` 은 디버그 빌드에서 이중 호출·미호출을 감지해 경고해주므로, `withUnsafeThrowingContinuation` 보다 먼저 이걸로 시작하는 걸 권한다. 성능이 진짜 문제가 되는 게 확인된 지점에서만 unsafe 버전으로 바꿔라.

이미 URLSession을 쓰고 있다면 `URLSession.shared.data(for:)` 같은 네이티브 async API로 아예 갈아타는 것도 방법이다. 다만 커스텀 인터셉터·리트라이 로직이 completion 기반으로 짜여 있다면, 그 로직까지 한 번에 옮기려 하지 말고 먼저 continuation으로 감싸서 인터페이스만 async로 바꾸는 걸 1차 목표로 잡는 게 리스크가 작다.

## 4. 3단계 — UI 갱신: `@MainActor`

여기서부터 **"타이밍이 미묘하게 달라지는" 진짜 위험 구간**이 시작된다.

```swift
// Before
func updateLabel(_ text: String) {
    DispatchQueue.main.async {
        self.label.text = text
    }
}

// After
@MainActor
func updateLabel(_ text: String) {
    label.text = text
}
```

기능적으로는 같아 보이지만 결정적인 차이가 있다.

- `DispatchQueue.main.async`는 **호출 시점이 메인 스레드여도 무조건 큐에 넣고, 현재 런루프 사이클이 끝난 뒤 실행**된다.
- `@MainActor` 함수를 이미 메인 액터 컨텍스트에서 호출하면 **큐잉 없이 즉시(동기적으로) 실행**될 수 있다. (Swift 런타임이 이미 같은 실행자(executor) 위에 있으면 홉을 생략한다.)

이 차이 때문에 "다음 런루프까지 기다렸다가 실행되는 걸 전제로" 짜인 코드(예: 애니메이션 completion 직후 상태 리셋, 레이아웃 강제 갱신 순서)가 실행 순서를 바꿔버릴 수 있다. 그래서 이 단계는 **화면 단위로 쪼개서, 각 화면을 옮긴 뒤 반드시 실기기/시뮬레이터로 애니메이션·스크롤·회전 같은 타이밍에 민감한 인터랙션을 직접 눌러보고 넘어가야 한다.**

> **노하우**: ViewController/View 전체를 `@MainActor` 로 한 번에 선언하지 말고, UI를 직접 만지는 메서드 단위로 먼저 붙여라. 클래스 전체에 `@MainActor` 를 붙이면 그 안의 모든 메서드(백그라운드에서 불려야 하는 것까지)가 메인 액터로 격리돼, 의도치 않게 백그라운드 작업까지 메인 스레드로 끌려온다.

## 5. 4단계 — fire-and-forget 작업

`DispatchQueue.global().async { ... }` 처럼 결과를 기다리지 않는 백그라운드 작업은 `Task { ... }` 로 1:1 치환이 가능하지만, **취소와 캡처를 반드시 같이 점검해야 한다.**

```swift
// Before
func startPolling() {
    DispatchQueue.global().async {
        while true {
            self.fetchLatest()
            Thread.sleep(forTimeInterval: 5)
        }
    }
}

// After
private var pollingTask: Task<Void, Never>?

func startPolling() {
    pollingTask = Task {
        while !Task.isCancelled {
            await fetchLatest()
            try? await Task.sleep(for: .seconds(5))
        }
    }
}

func stopPolling() {
    pollingTask?.cancel()
}
```

GCD는 "한번 던진 작업을 중간에 멈춘다"는 개념이 약했다(`DispatchWorkItem.cancel()`이 있지만 실행 중인 클로저 내부까지 끊어주진 않는다). Swift Concurrency의 `Task` 취소는 **협조적**이다 — `Task.cancel()`을 호출해도 작업이 스스로 `Task.isCancelled` 나 `Task.checkCancellation()` 을 확인하지 않으면 계속 돈다. GCD 코드를 그대로 옮기면서 취소 체크를 안 넣는 게 이 단계에서 가장 흔한 실수다.

> **함정**: `Task { self.doSomething() }` 처럼 `self`를 암묵적으로 강하게 캡처하는 습관은 GCD 클로저 캡처 습관을 그대로 옮긴 것이다. 화면이 사라져도 Task가 살아있는 동안 `self`가 해제되지 않는다. `deinit`에서 `task?.cancel()` 을 명시적으로 호출하거나, `[weak self]` 캡처 + 조기 return을 습관화해야 한다.

## 6. 5단계 — `DispatchGroup` → `async let` / `TaskGroup`

여러 비동기 작업을 병렬로 돌리고 다 끝나면 합치는 패턴은 구조적 동시성(structured concurrency)으로 옮기면 코드가 눈에 띄게 단순해진다.

```swift
// Before
func loadDashboard(completion: @escaping (Profile, [Post], Notifications) -> Void) {
    var profile: Profile!
    var posts: [Post]!
    var notifications: Notifications!
    let group = DispatchGroup()

    group.enter(); fetchProfile { profile = $0; group.leave() }
    group.enter(); fetchPosts { posts = $0; group.leave() }
    group.enter(); fetchNotifications { notifications = $0; group.leave() }

    group.notify(queue: .main) {
        completion(profile, posts, notifications)
    }
}

// After
func loadDashboard() async throws -> (Profile, [Post], Notifications) {
    async let profile = fetchProfile()
    async let posts = fetchPosts()
    async let notifications = fetchNotifications()
    return try await (profile, posts, notifications)
}
```

개수가 정해져 있으면 `async let`, 개수가 동적(배열 순회 등)이면 `TaskGroup` 을 쓴다.

```swift
func fetchAllThumbnails(ids: [String]) async throws -> [UIImage] {
    try await withThrowingTaskGroup(of: (Int, UIImage).self) { group in
        for (index, id) in ids.enumerated() {
            group.addTask { (index, try await self.fetchThumbnail(id: id)) }
        }
        var results = [Int: UIImage]()
        for try await (index, image) in group {
            results[index] = image
        }
        return ids.indices.map { results[$0]! }
    }
}
```

> **노하우**: `DispatchGroup` 버전은 `var profile: Profile!` 처럼 옵셔널 강제 언래핑 변수를 캡처하는 패턴이 흔했는데, 이건 컴파일러가 "초기화 전에 쓰였는지" 검증해주지 못한다. `async let` / `TaskGroup` 은 값을 리턴 경로로 다루기 때문에 이런 종류의 초기화 누락 버그가 컴파일 타임에 걸러진다. 마이그레이션 중 이런 버그를 몇 개는 발견하게 될 것이다 — 원래 있던 버그다.

## 7. 6단계 — Actor: 여기서부터는 진짜 신중하게

serial `DispatchQueue` 로 상호배제하던 공유 상태(캐시, 카운터, 커넥션 풀 등)를 `Actor` 로 옮기는 단계다. 겉보기엔 가장 "Swift Concurrency다운" 리팩터지만, **동작 방식이 근본적으로 다르기 때문에 8단계 중 가장 리스크가 크다.**

```swift
// Before
final class ImageCache {
    private let queue = DispatchQueue(label: "image-cache")
    private var storage: [String: UIImage] = [:]

    func image(for key: String) -> UIImage? {
        queue.sync { storage[key] }
    }

    func store(_ image: UIImage, for key: String) {
        queue.async { self.storage[key] = image }
    }
}

// After
actor ImageCache {
    private var storage: [String: UIImage] = [:]

    func image(for key: String) -> UIImage? {
        storage[key]
    }

    func store(_ image: UIImage, for key: String) {
        storage[key] = image
    }
}
```

serial queue는 **"락"처럼 동작한다** — `queue.sync` 로 들어간 코드는 그 사이 다른 코드가 절대 끼어들지 못한다. 반면 actor는 **재진입(reentrant)** 이 가능하다. actor 메서드 안에 `await` 지점이 있으면, 그 지점에서 다른 호출자가 같은 actor의 다른 메서드를 먼저 실행하고 상태를 바꿔놓을 수 있다.

```swift
actor Account {
    private var balance = 0

    func deposit(_ amount: Int) async {
        let current = balance
        await someAsyncValidation()   // ← 이 await 지점에서 다른 deposit()이 끼어들 수 있다
        balance = current + amount    // 그 사이 바뀐 balance를 덮어써버림 — 경쟁 상태
    }
}
```

GCD serial queue를 그대로 actor로 치환하면 컴파일은 되지만, **await 지점을 사이에 두고 상태를 읽고 나중에 쓰는 패턴**이 있으면 미묘한 경쟁 상태가 생긴다. 해결책은 "읽고-계산하고-쓰기"를 await 지점 없이 한 번에 처리하거나, 값을 지역 변수로 스냅샷 떠서 검증 로직에 넘기는 식으로 재작성하는 것이다.

> **노하우**: actor로 옮길 때는 **"이 메서드 안에 await가 몇 개 있고, 그 사이에 상태를 다시 읽는 곳이 있는가"** 를 한 줄씩 체크리스트로 만들어라. 없으면 안전하게 치환된 것이고, 있으면 리팩터가 필요하다는 신호다. 이 단계는 자동 변환 도구(Xcode의 concurrency 마이그레이터)를 참고용으로만 쓰고, 결과를 반드시 사람이 검토해야 한다.

## 8. 7단계 — Combine/콜백 체인 → `AsyncSequence`

Combine으로 짜인 스트림이나, 콜백이 여러 번 불리는 델리게이트 패턴(예: 위치 업데이트, 소켓 메시지 수신)은 `AsyncSequence`로 옮기면 `for await` 로 순회할 수 있다.

```swift
final class LocationStream {
    private var continuation: AsyncStream<CLLocation>.Continuation?

    lazy var updates: AsyncStream<CLLocation> = AsyncStream { continuation in
        self.continuation = continuation
        locationManager.startUpdatingLocation()
    }

    func didUpdate(_ location: CLLocation) {
        continuation?.yield(location)
    }

    func stop() {
        continuation?.finish()
        locationManager.stopUpdatingLocation()
    }
}

// 사용부
for await location in locationStream.updates {
    updateMap(with: location)
}
```

이 단계는 화면 하나하나가 이미 async/await, @MainActor, Task 취소에 익숙해진 뒤에 손대는 게 좋다. 왜냐하면 `for await` 루프는 그 자체로 `Task` 안에서 돌아야 하고, 화면이 사라질 때 그 Task를 취소해줘야 스트림이 정리되기 때문이다 — 앞선 4단계(fire-and-forget)의 취소 습관이 안 잡혀 있으면 여기서 메모리 누수가 난다.

---

## 9. 8단계 — Strict Concurrency Checking: 마지막 관문

여기까지 왔다면 프로젝트 대부분이 async/await·Actor 기반으로 돌아가고 있을 것이다. 마지막은 컴파일러의 데이터 경쟁 검증을 켜는 단계다. Xcode 빌드 세팅의 `SWIFT_STRICT_CONCURRENCY` 를 단계적으로 올린다.

```
minimal   → 명시적으로 표시한 코드만 검사 (기본값, 사실상 거의 검사 안 함)
targeted  → async/await·Actor를 이미 쓰는 코드만 엄격 검사
complete  → 프로젝트 전체를 Swift 6 데이터 경쟁 안전성 기준으로 검사
```

`minimal → targeted` 로 올리는 순간부터 `Sendable` 을 만족하지 않는 타입을 actor 경계 너머로 넘기는 코드에 경고가 쏟아진다. 이때가 되어서야 비로소 "지금까지 옮긴 코드 중 실제로 스레드 안전하지 않은 부분"이 드러난다 — 즉, **strict concurrency checking은 마이그레이션의 사후 검증 도구이지, 처음부터 켜고 시작할 도구가 아니다.** 처음부터 `complete` 로 켜면 경고 수백 개에 압도되어 손을 놓게 된다.

> **노하우**: `targeted` 모드에서 새로 생기는 경고는 파일 단위로 `@preconcurrency import` 나 개별 `@unchecked Sendable` 로 임시 억제해 두고 백로그로 관리하되, **새로 작성하는 코드에는 억제 어노테이션을 쓰지 않는다**는 팀 규칙을 먼저 세워라. 그래야 "이미 있던 부채"와 "새로 짜는 코드"가 뒤섞이지 않는다.

## 10. 요약 — 단계별 체크리스트

| 단계 | 대상 | 핵심 리스크 | 검증 포인트 |
|---|---|---|---|
| 1 | 순수 함수/유틸리티 | 거의 없음 | 단위 테스트 |
| 2 | 네트워크 completion → async | continuation 이중 호출 | `withCheckedThrowingContinuation` 경고 유무 |
| 3 | UI 갱신 `@MainActor` | 실행 타이밍(즉시 vs 다음 런루프) 변화 | 애니메이션·스크롤 실기기 테스트 |
| 4 | fire-and-forget → `Task` | 취소 누락, `self` 강캡처 | `deinit`에서 취소 확인 |
| 5 | `DispatchGroup` → `async let`/`TaskGroup` | (오히려 버그 발견 가능) | 옵셔널 강제 언래핑 제거 확인 |
| 6 | serial queue → `Actor` | 재진입으로 인한 경쟁 상태 | await 지점 사이 상태 재사용 여부 |
| 7 | Combine/콜백 → `AsyncSequence` | 스트림 미해제 누수 | 화면 dismiss 시 Task 취소 확인 |
| 8 | Strict Concurrency Checking | 경고 폭증 | targeted → complete 단계적 상향 |

Swift Concurrency 전환은 "문법을 바꾸는 일"이 아니라 **"동시성 모델 자체를 바꾸는 일"** 이다. 문법만 보고 기계적으로 치환하면 컴파일은 통과하지만 타이밍과 상호배제 방식이 조용히 달라진다. 리스크가 낮은 leaf 코드부터 시작해서, 상태 공유가 얽힌 actor·strict concurrency는 프로젝트 전체가 async/await에 충분히 익숙해진 마지막에 손대는 순서를 권한다.
