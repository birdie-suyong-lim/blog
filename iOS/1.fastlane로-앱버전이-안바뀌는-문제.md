> 작성일: 2026-06-18
> 주제: iOS 릴리즈 자동화 / fastlane / Xcode 버전 관리

릴리즈를 준비하면서 늘 쓰던 `fastlane setVersion` 으로 앱 버전을 올렸다. 명령은 "성공 🎉" 으로 끝났는데, 막상 프로젝트를 열어 보니 **마케팅 버전이 그대로**였다. 빌드 번호는 1로 리셋됐는데 버전만 안 바뀐, 묘하게 "절반만 동작한" 상태였다.

이 글은 그 원인을 추적해 **왜 `agvtool` 기반 버전 변경이 무력화됐는지**, 그리고 **어떻게 안정적으로 고쳤는지**를 정리한 기록이다. 같은 함정(자동 생성 Info.plist 환경에서 버전 자동화가 조용히 실패)을 밟을 수 있는 분들을 위한 글이다.

---

## 0. 배경 지식 — iOS 앱의 "버전"은 어디에 저장되는가

먼저 용어를 정리하자. iOS 앱에는 사용자에게 보이는 두 종류의 버전이 있다.

| 표시 | Info.plist 키 | Xcode 빌드 세팅 | 의미 |
|---|---|---|---|
| 마케팅 버전 (1.10.0) | `CFBundleShortVersionString` | `MARKETING_VERSION` | App Store에 노출되는 사람이 읽는 버전 |
| 빌드 번호 (42) | `CFBundleVersion` | `CURRENT_PROJECT_VERSION` | 같은 마케팅 버전 안에서 증가하는 내부 빌드 식별자 |

여기서 핵심은 **"같은 값을 두 군데(Info.plist / 빌드 세팅)에서 표현할 수 있다"** 는 점이다. 그리고 이 둘의 관계는 Xcode 프로젝트 설정에 따라 달라진다.

### 옛날 방식 vs 요즘 방식

- **옛날 방식 (수동 Info.plist):** 프로젝트에 물리적인 `Info.plist` 파일이 있고, 그 안에 `CFBundleShortVersionString = 1.10.0` 같은 값이 **문자열로 직접** 박혀 있다.
- **요즘 방식 (자동 생성):** Xcode 13부터 도입된 `GENERATE_INFOPLIST_FILE = YES` 설정. 이걸 켜면 Xcode가 **빌드 시점에 Info.plist를 자동으로 합성**한다. 버전 값은 `MARKETING_VERSION` / `CURRENT_PROJECT_VERSION` **빌드 세팅에서 주입**되고, 소스 트리의 Info.plist에는 `CFBundleShortVersionString` 키 자체가 **존재하지 않는다.**

즉 요즘 방식에서 버전의 **단일 진실 소스(Single Source of Truth)는 `.xcodeproj` 안의 빌드 세팅**이다. Info.plist는 결과물일 뿐이다. 이 차이가 이번 이슈의 출발점이다.

## 1. 이슈 발생 원인

### 증상

릴리즈 자동화 스크립트(fastlane lane)는 대략 이렇게 생겼었다.

```ruby
# 문제가 된 lane
lane :setVersion do |values|
  increment_version_number(version_number: values[:v])   # 마케팅 버전 변경
  increment_build_number(build_number: 1)                # 빌드 번호 1로 리셋
end
```

`fastlane setVersion v:1.10.0` 을 실행하면:

- ✅ 빌드 번호 → 1로 리셋됨 (정상)
- ❌ 마케팅 버전 → **이전 버전 그대로** (실패)
- 그런데도 fastlane은 **에러 없이 "성공"으로 종료**

가장 위험한 종류의 버그다. **실패했는데 실패처럼 보이지 않는다.** 빌드 번호는 멀쩡히 바뀌니 로그만 보면 "버전 작업이 다 된 것 같은" 착시가 생긴다.

### 실행 로그를 들여다보니

`increment_version_number` 액션이 내부에서 실제로 호출한 명령은 이거였다.

```
$ agvtool new-marketing-version 1.10.0
▸ Setting CFBundleShortVersionString of project ...
▸ Updating CFBundleShortVersionString in Info.plist(s)...
▸ Updated CFBundleShortVersionString in ".../App/Info.plist" to 1.10.0
```

로그상으로는 **"Info.plist를 1.10.0으로 업데이트했다"** 고 당당히 말한다. 그런데 정작 `.xcodeproj` 의 `MARKETING_VERSION` 빌드 세팅은 꿈쩍도 안 했고, git diff에도 Info.plist 변경은 잡히지 않았다.

여기서 원인이 드러난다.

> **`agvtool new-marketing-version` 은 빌드 세팅이 아니라 Info.plist의 `CFBundleShortVersionString` 키를 수정한다.**
> 그런데 이 프로젝트는 `GENERATE_INFOPLIST_FILE = YES` 라 **소스 Info.plist에 그 키가 아예 없다.**
> 결과적으로 agvtool은 "존재하지 않는 키"를 향해 헛스윙을 했고, **진짜 버전 소스인 `MARKETING_VERSION` 빌드 세팅은 건드리지 못했다.**

## 2. 이슈 해설 — 왜 빌드 번호는 됐는데 버전만 안 됐을까

여기서 자연스러운 의문이 든다. **같은 agvtool인데 왜 빌드 번호(`increment_build_number`)는 정상 동작했을까?**

핵심은 **fastlane 액션마다 "어디에 쓰는지"가 다르다**는 데 있다.

- `increment_build_number` 에 `xcodeproj:` 인자를 주면, 이 액션은 **빌드 세팅(`CURRENT_PROJECT_VERSION`)을 직접** 읽고 쓴다. → 자동 생성 Info.plist 환경에서도 정상 동작.
- 반면 `increment_version_number` 는 `agvtool new-marketing-version` 에 의존하는데, agvtool은 **Info.plist의 키**를 타깃으로 한다. → 그 키가 없으면 무력화.

그래서 "절반만 성공"이라는 기묘한 증상이 나온 것이다. 빌드 번호 경로는 빌드 세팅을 직접 만지니 멀쩡했고, 마케팅 버전 경로는 빈 Info.plist를 향해 헛질을 한 것.

### 왜 예전엔 (아마) 잘 됐던 것처럼 보였나

이런 자동화는 보통 "예전엔 됐는데?" 라는 기억과 함께 온다. 두 가지 가능성이 있다.

1. **원래부터 안 됐는데 아무도 몰랐다.** 과거 버전 bump 커밋이 사실은 사람이 수동으로 빌드 세팅을 고친 것이고, 자동화는 줄곧 헛스윙만 하고 있었을 수 있다. (빌드 번호 리셋이 같이 일어나니 "동작한 것처럼" 보인다.)
2. **프로젝트가 중간에 자동 생성 방식으로 마이그레이션됐다.** 과거엔 수동 Info.plist라 agvtool이 먹혔지만, `GENERATE_INFOPLIST_FILE = YES` 로 전환되면서 키가 사라졌고, 그때부터 조용히 깨졌다.

어느 쪽이든 교훈은 같다. **"성공 메시지"는 "의도한 결과"를 보증하지 않는다.** 자동화는 반드시 *결과 상태*를 검증해야 한다.

## 3. 이슈 해결 방법

해결의 방향은 명확하다. **버전의 진짜 소스인 `MARKETING_VERSION` 빌드 세팅을 직접 수정하면 된다.** agvtool이라는 우회로를 버리고 소스를 정조준한다.

### 선택지 비교

| 방법 | 장점 | 단점 |
|---|---|---|
| agvtool (`increment_version_number`) | 표준, 한 줄 | **자동 생성 Info.plist 환경에서 무력화** |
| 빌드 세팅 텍스트 치환 (sed/정규식) | 의존성 없음 | `.pbxproj` 정규식은 깨지기 쉬움, 타깃 구분 어려움 |
| **`xcodeproj` gem 으로 빌드 세팅 직접 수정** | 구조적·결정적, 타깃 단위 제어 | gem 의존 (단, fastlane에 이미 번들됨) |

세 번째를 택했다. fastlane은 내부적으로 [`xcodeproj`](https://github.com/CocoaPods/Xcodeproj) gem을 이미 포함하고 있어서 **추가 설치 없이** `.xcodeproj` 를 객체로 열어 빌드 세팅을 조작할 수 있다.

### 고친 코드

```ruby
def setVersionNumber(version)
  version = version.to_s.strip
  UI.user_error!("버전 미지정 — 예: fastlane setVersion v:1.10.0") if version.empty?

  # GENERATE_INFOPLIST_FILE = YES 프로젝트라 버전의 단일 소스는 빌드 세팅이다.
  # 소스 Info.plist에는 CFBundleShortVersionString 키 자체가 없어서
  # agvtool(increment_version_number)은 빌드 세팅을 바꾸지 못한다.
  # → xcodeproj gem으로 MARKETING_VERSION을 직접 기록한다.
  require 'xcodeproj'
  project = Xcodeproj::Project.open(File.expand_path('../App.xcodeproj', __dir__))

  updated = []
  project.native_targets.each do |target|
    # 테스트 번들은 앱 버전과 무관 — 제외
    next if target.test_target_type?

    target.build_configurations.each do |config|
      next unless config.build_settings.key?('MARKETING_VERSION')
      config.build_settings['MARKETING_VERSION'] = version
      config.build_settings['CURRENT_PROJECT_VERSION'] = '1'  # 새 버전 → 빌드 번호 1로 리셋
      updated << "#{target.name}/#{config.name}"
    end
  end
  project.save

  UI.user_error!("MARKETING_VERSION을 가진 앱 타깃을 찾지 못함") if updated.empty?
  UI.success("버전 #{version} (빌드번호 1) 설정 — #{updated.length}개 컨피그")
end
```

### 설계에서 신경 쓴 포인트 3가지

**① 테스트 타깃 제외 (`test_target_type?`)**
한 프로젝트에는 앱 본체 외에도 유닛 테스트, UI 테스트, 확장(extension) 등 여러 타깃이 있다. 테스트 번들의 버전(보통 `1.0` 으로 박혀 있다)까지 끌어올리면 안 된다. `xcodeproj` gem이 제공하는 `target.test_target_type?` 로 테스트 번들을 깔끔하게 걸러낸다. 그 결과 **앱 본체 + 확장 + 빌드 변형(dev/prod)** 만 정확히 잡혀 올라간다.

**② 빌드 번호 리셋을 같은 패스에서 처리**
새 마케팅 버전이 나오면 빌드 번호는 1부터 다시 센다. `MARKETING_VERSION` 을 쓰는 김에 `CURRENT_PROJECT_VERSION = 1` 도 같은 루프에서 기록해, 빌드 세팅 한 번의 저장으로 둘 다 결정적으로 맞춘다. (agvtool 왕복 없음.)

**③ 가드와 검증 메시지**
빈 버전이 들어오면 즉시 중단(`UI.user_error!`)하고, 매칭된 타깃이 0개면 "조용한 성공"을 막기 위해 에러를 던진다. 마지막에 **실제로 바뀐 컨피그 개수**를 출력해, 다음부터는 로그만 봐도 "몇 개가 바뀌었는지" 확인할 수 있게 했다.

### 검증

고친 다음에는 반드시 *결과 상태*를 직접 확인해야 한다. 이번 버그의 교훈이 정확히 그거였으니까.

```bash
# 1) 마케팅 버전이 실제로 빌드 세팅에 박혔는지
$ grep "MARKETING_VERSION = 1.10.0" App.xcodeproj/project.pbxproj | wc -l
6      # 앱 본체 + 확장 + dev/prod × Debug/Release = 6개 컨피그

# 2) 테스트 타깃은 건드리지 않았는지 (1.0 유지)
$ grep "MARKETING_VERSION = 1.0" App.xcodeproj/project.pbxproj | wc -l
4      # 테스트 번들 2종 × Debug/Release

# 3) 프로젝트 파일이 여전히 유효한지 (파싱 + 타깃 목록)
$ xcodebuild -list -project App.xcodeproj
```

추가로, 같은 명령을 **두 번 연속 실행**해서 두 번째엔 변경이 없는지(=멱등성) 확인했다. 자동화는 여러 번 돌려도 같은 결과여야 안심하고 쓸 수 있다.

> ⚠️ 한 가지 주의: `xcodeproj` gem으로 `.pbxproj` 를 저장하면 빈 배열 표기 등 일부 포맷을 자기 스타일로 **정규화**한다(특히 파일시스템 동기화 그룹을 쓰는 프로젝트에서 두드러진다). 기능상 무해하고 한 번 저장되면 그 뒤로는 멱등하지만, 첫 적용 시 diff에 버전과 무관한 줄이 섞여 들어올 수 있으니 코드 리뷰 때 인지하고 있는 게 좋다. diff를 극단적으로 깨끗하게 유지해야 한다면 빌드 세팅 줄만 정조준하는 텍스트 치환을 고려할 수도 있다.

## 4. 회고 — 무엇을 배웠나

이번 건의 본질은 사실 버전 한 줄이 아니라 **"자동화의 신뢰성"** 에 대한 이야기다.

1. **추상화의 전제를 의심하라.** `increment_version_number` 는 "버전을 올린다"는 추상화를 제공하지만, 그 밑에는 "Info.plist의 특정 키를 고친다"는 구체적 전제가 깔려 있다. 프로젝트가 그 전제(`GENERATE_INFOPLIST_FILE = YES`)를 벗어나는 순간 추상화는 조용히 깨진다.
2. **"성공 메시지" ≠ "의도한 결과".** 종료 코드 0과 🎉 이모지는 *명령이 끝났다*는 뜻일 뿐, *원하는 상태가 됐다*는 보증이 아니다. 자동화에는 항상 결과 상태에 대한 사후 검증(assert)을 붙이자.
3. **단일 진실 소스를 직접 겨냥하라.** 같은 데이터를 두 군데서 표현할 수 있을 때(Info.plist 키 vs 빌드 세팅), 우회 도구가 어느 쪽을 만지는지 모호하면 사고가 난다. 진짜 소스를 알아내고 거기를 직접 쓰는 편이 결국 더 단순하고 안전하다.

같은 증상(fastlane이 "성공"했는데 버전이 안 바뀐다)을 만났다면, 먼저 **`GENERATE_INFOPLIST_FILE` 설정과 소스 Info.plist에 `CFBundleShortVersionString` 키가 실제로 있는지**부터 확인해 보길 권한다. 거기서 거의 다 갈린다.
