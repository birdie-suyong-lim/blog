# 팀 레포에서 함께 바이브코딩하기 — 협업용 클로드 초기 세팅

> 작성일: 2026-05-19
> 출처: Claude Code 공식 문서 (https://code.claude.com/docs) + 본 위키 운영 경험
> 형식: 협업 repository에 Claude Code 초기 세팅을 까는 실전 가이드

혼자 쓰는 Claude Code는 세팅이 망가져도 나만 불편하다. 하지만 **팀이
같은 repository에서 함께 바이브코딩**하면 이야기가 달라진다. 각자
다른 컨벤션으로 AI에게 코드를 뽑게 두면, 같은 레포 안에서 스타일이
갈라지고, 리뷰가 누락되고, 누군가의 개인 권한 설정이 사고를 부른다.

핵심은 하나다. **세팅 자체를 레포에 코드처럼 커밋해서, 누가 clone해도
같은 가드레일 위에서 AI가 움직이게 만드는 것.** 이 글은 그 초기 세팅을
"무엇을 공유하고 무엇을 개인 것으로 둘지"부터 단계별로 정리한다.

---

## 1. 가장 중요한 원칙 — 공유할 것 vs 개인 것

팀 세팅의 모든 결정은 이 한 질문으로 환원된다.
**"이건 레포에 커밋되어 모두에게 강제돼야 하는가, 아니면 내 머신에만
있어야 하는가?"**

| 성격 | 파일/대상 | 커밋? |
|------|----------|-------|
| 팀 공용 규칙 | `CLAUDE.md` (레포 루트) | ✅ 커밋 |
| 팀 공용 권한·훅·환경 | `.claude/settings.json` | ✅ 커밋 |
| 팀 공용 슬래시 명령 | `.claude/commands/` | ✅ 커밋 |
| 팀 공용 스킬 | `.claude/skills/` | ✅ 커밋 |
| 팀 공용 서브에이전트 | `.claude/agents/` | ✅ 커밋 |
| 팀 공용 MCP 서버 | `.mcp.json` (레포 루트) | ✅ 커밋 |
| **개인 권한 오버라이드** | `.claude/settings.local.json` | ❌ gitignore |
| **개인 토큰·시크릿** | `.env`, MCP 인증값 | ❌ gitignore |
| **개인 메모리** | `~/.claude/.../memory/` | ❌ 애초에 레포 밖 |
| **개인 글로벌 설정** | `~/.claude/CLAUDE.md`, `~/.claude/settings.json` | ❌ 레포 밖 |

> 한 줄 요약: **`.claude/settings.json`은 팀 헌법, `settings.local.json`은
> 내 방 인테리어.** 전자는 커밋, 후자는 절대 커밋 금지.

메모리(자동 메모리 시스템)는 **개인 컨텍스트**라 팀과 공유되지 않는다.
팀이 공유해야 할 지식은 메모리가 아니라 **CLAUDE.md·skills·commands**로
박아야 한다. 이 구분을 놓치면 "나는 분명 알려줬는데 동료 AI는 모른다"가
반복된다.

## 2. .claude 디렉토리 — 무엇을 커밋하고 무엇을 gitignore할까

협업 레포의 `.claude/` 권장 구조:

```
<repo>/
├── CLAUDE.md                    # 팀 공용 규칙 (커밋)
├── .mcp.json                    # 팀 공용 MCP 서버 (커밋)
├── .claude/
│   ├── settings.json            # 팀 공용 권한·훅·env (커밋)
│   ├── settings.local.json      # 개인 오버라이드 (gitignore)
│   ├── commands/                # 팀 슬래시 명령 (커밋)
│   │   └── deploy-check.md
│   ├── skills/                  # 팀 스킬 (커밋)
│   ├── agents/                  # 팀 서브에이전트 (커밋)
│   └── ONBOARDING.md            # 신규 합류 가이드 (커밋, 선택)
└── .gitignore
```

`.gitignore`에 반드시 들어가야 할 항목:

```gitignore
# Claude Code — 개인 전용, 커밋 금지
.claude/settings.local.json
.claude/*.local.json

# 시크릿
.env
.env.local
```

> Claude Code는 `settings.local.json`을 만들 때 자동으로 gitignore에
> 추가해 주지만, **레포 초기화 시점에 위 항목을 명시적으로 박아두는 게
> 안전하다.** 한 명이라도 구버전 클라이언트를 쓰면 개인 권한이 통째로
> 커밋돼 팀에 새어 나갈 수 있다.

## 3. CLAUDE.md — 팀 공용 규칙의 헌법

`CLAUDE.md`는 매 요청마다 모든 팀원의 AI에게 전달되는 **공통 시스템
규칙**이다. 바이브코딩에서 컨벤션이 갈라지지 않게 막는 1차 방어선.

`/init` 슬래시 명령으로 코드베이스를 스캔해 초안을 만든 뒤, 다음을
손으로 다듬는다.

**팀 CLAUDE.md에 꼭 들어갈 것:**

- **빌드·테스트·린트 명령** — AI가 검증할 때 쓸 정확한 커맨드
  (`npm run test`, `make lint` 등)
- **디렉토리 구조와 책임** — 어디에 무엇을 두는지
- **코딩 컨벤션** — 네이밍, 에러 처리, 로깅, 주석 밀도
- **하지 말 것 (금지 목록)** — 직접 prod 접근 금지, 특정 라이브러리
  금지, 시크릿 하드코딩 금지 등
- **PR·커밋 규칙** — 브랜치 전략, 커밋 메시지 포맷, 리뷰 게이트
- **AI 작업 프로토콜** — "큰 변경은 plan mode로 계획 승인 후 진행",
  "PR 전 `/review` 필수" 같은 워크플로우 규약

**팀 CLAUDE.md에 넣지 말 것:**

- 개인 취향 (내 에디터 설정, 내 호칭 규칙)
- 일회성 메모 (그건 메모리나 별도 노트로)
- 장황한 코드 예시 — 커지면 `@docs/conventions.md` 처럼 임포트로 분리

> CLAUDE.md는 **짧고 불변인 정책만**. 비대해지면 모든 팀원의 매 요청이
> 느려진다. 다이어트 원칙은 `article/AI/3.내-클로드-빠르게-만들기.md`
> 참조. 글로벌·프로젝트 중복 정리는 `article/AI/2.ai-정책파일-관리.md`.

## 4. settings.json — 팀 공용 권한·훅·환경

`.claude/settings.json`은 커밋되어 **모든 팀원에게 동일하게 적용**되는
권한·훅·환경의 집합이다. 바이브코딩의 안전벨트가 여기 들어간다.

### A. 권한 — 안전한 건 자동 허용, 위험한 건 차단

```jsonc
{
  "permissions": {
    "allow": [
      "Bash(npm run test:*)",
      "Bash(npm run lint:*)",
      "Bash(npm run build)",
      "Bash(git status)",
      "Bash(git diff:*)",
      "Read"
    ],
    "deny": [
      "Bash(git push --force:*)",
      "Bash(rm -rf:*)",
      "Bash(npm publish)",
      "Read(./.env)",
      "Read(./secrets/**)"
    ]
  }
}
```

- **allow**: 빌드·테스트·린트 같은 안전한 read-only/검증 명령을 자동
  허용 → 매번 권한 prompt 뜨는 마찰 제거 (바이브코딩 흐름 유지)
- **deny**: `deny`는 `allow`를 **항상 이긴다.** force push, prod 배포,
  시크릿 읽기 같은 돌이킬 수 없는 행위를 팀 차원에서 원천 차단
- 개인이 추가로 허용하고 싶은 건 각자 `settings.local.json`에서
  (팀 설정을 건드리지 않고 개인만 확장)

### B. 훅 — 사람이 신경 안 써도 컨벤션이 강제되게

훅은 바이브코딩에서 **컨벤션 드리프트를 막는 핵심**이다. AI가 코드를
아무리 빨리 뽑아도, 훅이 자동으로 포맷·린트·테스트를 돌려 품질선을 유지.

```jsonc
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [
          { "type": "command",
            "command": "npx prettier --write \"$CLAUDE_FILE_PATHS\" 2>/dev/null || true" }
        ]
      }
    ]
  }
}
```

- **PostToolUse (Edit/Write)**: 파일이 수정될 때마다 포매터 자동 적용
  → AI가 만든 코드도 팀 스타일로 정규화
- **Stop**: 세션 종료 시 테스트/타입체크 한 번 (선택, 무거우면 비동기)
- 훅은 **즉시 끝나는 가벼운 로직**으로. 무거운 훅은 전 팀원의 매 턴을
  느리게 만든다 (article 3 참고)

### C. 환경변수

```jsonc
{
  "env": {
    "NODE_ENV": "development"
  }
}
```

- 팀 공통 비-시크릿 환경변수만. **토큰·키는 절대 settings.json에 넣지
  말 것** (커밋되므로). 시크릿은 `settings.local.json`이나 `.env`로.

## 5. 바이브코딩 가드레일 — 팀에서 AI 코드가 무너지지 않게

여기가 협업 세팅의 핵심이다. 혼자면 막 짜도 되지만, 팀에서 여러 명이
동시에 AI로 코드를 쏟아내면 **일관성·리뷰·안전**이 가장 먼저 무너진다.
세팅으로 이걸 구조적으로 막는다.

| 위험 | 가드레일 | 세팅 위치 |
|------|---------|----------|
| 큰 변경을 무계획으로 질러버림 | "대규모 변경은 **plan mode**로 계획 승인 후 진행" 규약 | CLAUDE.md |
| 스타일이 사람마다 갈라짐 | PostToolUse 훅에 **자동 포맷/린트** | settings.json |
| 검증 없이 머지 | "PR 전 `/review`·`/security-review` 필수" | CLAUDE.md |
| 위험 명령 사고 | force push·prod·시크릿 **deny** | settings.json |
| 테스트 깨진 채 진행 | Stop 훅 또는 커밋 전 테스트 게이트 | settings.json |
| 컨벤션을 AI가 매번 잊음 | 컨벤션을 **CLAUDE.md에 명문화** | CLAUDE.md |
| 개인 설정이 팀에 샘 | `settings.local.json` gitignore | .gitignore |

핵심 원리: **마찰은 안전한 곳에서 0으로, 위험한 곳에서 최대로.**
테스트·린트는 자동 허용해 흐름을 끊지 않되, 되돌릴 수 없는 행위는
계획 승인·리뷰·deny로 반드시 사람을 거치게 한다. 바이브코딩의 속도와
팀의 안전을 동시에 잡는 균형점이 바로 이 표다.

## 6. 공유 자산 — commands · skills · subagents · MCP

팀이 자주 하는 작업은 **각자 프롬프트로 매번 치지 말고** 레포에 박아
공유한다.

- **`.claude/commands/`** — 슬래시 명령. 예: `/deploy-check`,
  `/pr-prep`. 마크다운 파일 하나 = 명령 하나. 커밋하면 전원이 같은
  명령을 같은 방식으로 사용
- **`.claude/skills/`** — 도메인 워크플로우를 캡슐화한 스킬. 팀 고유의
  검토·생성 절차를 표준화
- **`.claude/agents/`** — 서브에이전트. 보조 작업은 Sonnet/Haiku 모델로
  명시해 메인 컨텍스트·비용 보호 (메커니즘은
  `article/AI/1.클로드-스킬-에이전트-팀.md`)
- **`.mcp.json`** — 팀 공용 MCP 서버(이슈 트래커, 사내 DB 등). 단,
  **인증 토큰은 여기 넣지 말고** 환경변수/로컬로 주입

> 공유 자산도 비대화 대상이다. 안 쓰는 command·skill·agent는 즉시
> 삭제 — 카탈로그가 커지면 전 팀원의 도구 선택이 느려진다.

## 7. 온보딩 — clone하면 바로 같은 환경

세팅을 커밋하는 진짜 목적은 **신규 합류자가 clone 한 번으로 동일한
가드레일을 갖는 것**이다.

- 레포 루트 또는 `.claude/ONBOARDING.md`에 합류 절차를 적어둔다:
  필요한 CLI 버전, 시크릿을 `settings.local.json`/`.env`에 넣는 법,
  팀 슬래시 명령 목록, 작업 프로토콜 요약
- 신규자 체크리스트 예시:

```
☐ 레포 clone 후 .claude/settings.json 자동 적용 확인
☐ 개인 시크릿을 .claude/settings.local.json (또는 .env)에 주입
☐ MCP 인증 토큰 개인 주입 (.mcp.json은 서버 정의만, 토큰은 로컬)
☐ CLAUDE.md 정독 — 빌드/테스트 명령·금지목록·PR 규칙 숙지
☐ /review, /deploy-check 등 팀 슬래시 명령 한 번씩 시범 실행
☐ 첫 작업은 plan mode로 — 계획 승인 흐름 체험
```

> 잘 만든 팀 세팅의 척도: **새 팀원이 첫 PR을 올릴 때, 아무도 스타일·
> 컨벤션을 입으로 설명하지 않아도 되는 상태.** 그게 세팅이 코드처럼
> 작동한다는 증거다.

## 8. 초기 세팅 — 단계별 체크리스트

새 협업 레포에 Claude Code 세팅을 까는 순서:

```
1.  ☐ /init 으로 CLAUDE.md 초안 생성
2.  ☐ CLAUDE.md에 빌드/테스트/린트 명령·디렉토리 책임·금지목록 추가
3.  ☐ CLAUDE.md에 AI 작업 프로토콜 (plan mode·리뷰 게이트) 명문화
4.  ☐ .claude/settings.json 생성 — allow(테스트·린트) / deny(위험명령)
5.  ☐ settings.json에 PostToolUse 포맷/린트 훅 추가
6.  ☐ .gitignore에 settings.local.json·.env·*.local.json 추가
7.  ☐ 자주 쓰는 작업을 .claude/commands/ 슬래시 명령으로 추출
8.  ☐ 팀 공용 MCP 서버를 .mcp.json에 정의 (토큰은 제외)
9.  ☐ ONBOARDING.md 작성 — 신규 합류 절차·시크릿 주입법
10. ☐ settings.json/CLAUDE.md/.mcp.json/commands 커밋
11. ☐ settings.local.json이 커밋 목록에 없는지 git status로 최종 확인
12. ☐ 동료 한 명이 clone → 같은 권한·훅·명령 작동하는지 검증
```

11번과 12번이 가장 자주 빠지고, 가장 사고가 잦은 단계다.

## 9. 안티패턴 — 협업에서 흔한 함정

| 안티패턴 | 문제 | 해결 |
|----------|------|------|
| `settings.local.json`을 커밋 | 개인 권한·머신 설정이 팀 전체에 강제됨 | .gitignore에 명시, git status 확인 습관 |
| 토큰을 `settings.json`/`.mcp.json`에 하드코딩 | 시크릿이 레포 히스토리에 영구 노출 | env·local·시크릿 매니저로 주입 |
| 팀 지식을 메모리에 저장 | 메모리는 개인용 — 동료 AI는 그 지식을 모름 | CLAUDE.md·skills·commands에 박기 |
| 훅 없이 바이브코딩 | 사람마다 스타일 갈라짐, 컨벤션 드리프트 | PostToolUse 포맷/린트 훅 강제 |
| allow 범위를 너무 넓게 | `Bash(*)` 류 → AI가 위험 명령도 무프롬프트 실행 | 안전 명령만 화이트리스트, deny 우선 |
| CLAUDE.md에 개인 취향·일회성 메모 | 전 팀원 매 요청이 무거워짐 | 불변 팀 정책만, 나머진 분리/삭제 |
| 큰 변경을 plan mode 없이 질러버림 | 리뷰 불가능한 거대 diff, 방향 어긋남 | "대규모 변경은 계획 승인 후" 규약 |
| 리뷰 게이트 없이 머지 | AI가 만든 결함이 그대로 메인에 | PR 전 `/review`·`/security-review` 필수화 |

## 10. 정리

- 협업 세팅의 본질은 **"세팅을 코드처럼 커밋해 모두에게 같은 가드레일을
  강제"** 하는 것
- 분기점은 단 하나: **공유(`settings.json`·CLAUDE.md·commands)** vs
  **개인(`settings.local.json`·메모리·토큰)**
- 바이브코딩 안전은 **마찰의 비대칭 설계** — 안전한 검증은 자동 허용,
  되돌릴 수 없는 행위는 계획 승인·리뷰·deny
- 잘 된 세팅의 척도: **새 팀원이 clone 한 번으로 같은 규칙 위에서
  움직이고, 아무도 컨벤션을 입으로 설명하지 않아도 되는 상태**

> 30초 룰: 협업 레포에 손대기 전 `git status`로 **`settings.local.json`이
> staged에 없는지** 먼저 확인. 이 한 줄이 가장 흔한 사고를 막는다.

## 부록 — 프로젝트 초기 세팅 지시 프롬프트 (초안)

> 새 협업 레포에서 Claude Code에 아래를 그대로 붙여넣으면, 이 글의
> 원칙대로 초기 세팅을 잡아준다. `<...>` 자리는 알면 채우고, 모르면
> 비워둬도 된다 — Claude가 탐지하고 모호하면 질문하게 설계돼 있다.

```
역할: 너는 이 레포의 Claude Code 협업 세팅을 까는 담당이다.
목표는 "세팅을 코드처럼 커밋해 모든 팀원에게 같은 가드레일을
강제"하는 것이다. 속도(바이브코딩)와 안전을 동시에 잡아라.

[1] CLAUDE.md (커밋 대상) 작성/갱신
- 빌드·테스트·린트의 정확한 명령
- 디렉토리 구조와 각 영역의 책임
- 코딩 컨벤션 (네이밍·에러처리·로깅·주석 밀도)
- 금지 목록 (prod 직접 접근·시크릿 하드코딩·금지 라이브러리 등)
- PR·커밋·브랜치 규칙
- AI 작업 프로토콜: "대규모 변경은 plan mode로 계획 승인 후 진행",
  "PR 전 /review·/security-review 필수"
- 짧고 불변인 정책만. 길어지면 @docs/*.md 로 분리

[2] .claude/settings.json (커밋 대상) 작성/병합
- permissions.allow: 안전한 검증 명령만
  (테스트·린트·빌드·git status·git diff·Read)
- permissions.ask: force push, rm -rf, prod 배포/publish,
  .env·secrets 읽기
- hooks.PostToolUse (matcher "Edit|Write"): 포매터/린터 자동 적용,
  가볍게 끝나는 로직만 (무거우면 전 팀원 매 턴이 느려진다)
- env: 비-시크릿 공통 환경변수만. 토큰·키는 절대 금지

[3] .gitignore 갱신
- .claude/settings.local.json, .claude/*.local.json,
  .env, .env.local 추가

[4] 공유 자산
- 자주 반복할 작업을 .claude/commands/*.md 슬래시 명령으로
  추출할지 제안
- 팀 공용 MCP 서버가 있으면 .mcp.json 에 정의
  (인증 토큰은 절대 넣지 말 것)

[5] ONBOARDING.md (커밋 대상) 작성
- 필요한 CLI 버전, 시크릿을 settings.local.json/.env 에
  주입하는 법, 팀 슬래시 명령 목록, "첫 작업은 plan mode 권장"

[제약 — 반드시 지킨다]
- settings.local.json·토큰·시크릿은 절대 커밋하지 않는다
- 커밋 전 반드시 git status 를 보여주고,
  settings.local.json/.env 가 staged 에 없는지 내 확인을 받는다
- 커밋은 내가 명시적으로 지시할 때만 한다. 그 전엔 변경 요약만 보고
- 추측이 필요한 모든 지점(빌드 명령·컨벤션·금지목록·MCP)은
  진행 전에 질문한다
```

> 이 프롬프트는 **초안**이다. 팀 상황에 맞춰 `[1]`의 컨벤션·금지목록과
> `[2]`의 allow/deny 범위를 다듬어 쓰는 것을 전제로 한다.
