처음 쓸 땐 빠르고 똑똑하던 Claude Code가 몇 주~몇 달 지나면서 점점 느려지는 경험을 한 번씩 한다. 응답 지연, 권한 prompt 폭증, 매 요청마다 시작이 무거워지는 느낌. 대부분의 원인은 **단일한 병목이 아니라 누적된 비대화**다. 이 글은 느려지는 원인을 4개 카테고리로 분류하고, 각 영역에서 즉시 적용 가능한 최적화 방법을 정리한다.

---

## 1. 왜 점점 느려지는가 — 4가지 원인

```
                    [응답 시간]
                         │
       ┌──────────┬──────┴──────┬──────────┐
       │          │             │          │
   컨텍스트     세션          모델       인프라
   비대화      운영          선택       (캐시·I/O)
   ────────   ────────     ────────    ────────
   CLAUDE.md   /clear       Opus/      Prompt cache
   메모리       빈도         Sonnet/    파일 동기화
   에이전트     /compact     Haiku      MCP 서버
   히스토리     세션 길이    /fast      훅 무게
```

각 영역을 차례로 다룬다.

---

## 2. 컨텍스트 다이어트 — 가장 효과 큰 영역

응답 시간의 가장 큰 변수는 **매 요청마다 모델에게 보내는 토큰 양**이다. 모든 시스템 프롬프트, CLAUDE.md, 메모리, 도구 정의, 대화 히스토리가 매 턴마다 같이 간다. 토큰이 2배가 되면 응답도 비례적으로 느려진다.

### A. CLAUDE.md 슬리밍

| 점검 항목 | 액션 |
|----------|------|
| 안 쓰는 규칙·주석·예시 | 삭제 |
| 같은 정책이 글로벌·프로젝트 양쪽에 중복 | 한쪽만 남김 (이전 글 "AI 정책파일 관리" 참고) |
| 큰 코드 예시·테이블 | 별도 파일로 분리 + `@path/to/file.md` 임포트 |
| 더 이상 필요 없는 한시적 규칙 | 삭제 |

**측정**: `wc -l ~/.claude/CLAUDE.md` 와 `wc -l <repo>/CLAUDE.md` 합쳐서 200줄을 넘기 시작하면 다이어트 신호.

### B. 메모리 정리

자동 메모리 시스템이 활성화돼 있다면:

- **MEMORY.md 인덱스는 200줄 이내로 유지** (그 이상은 truncate됨, 정책상)
- 오래된 프로젝트 메모리·이미 outdated된 사실 정리
- 같은 사실 중복 메모리 통합

```bash
# 현재 메모리 크기 점검
wc -l ~/.claude/projects/*/memory/MEMORY.md
```

### C. 에이전트·스킬 정리

- 안 쓰는 `.claude/agents/*.md` 삭제 — 에이전트 정의는 description만 시스템 프롬프트에 노출되지만, 카탈로그가 커지면 모델의 도구 선택 부담이 늘어남
- `~/.claude/skills/` 와 프로젝트 `skills/` 양쪽에서 안 쓰는 것 정리
- `/init` 같은 시작 스킬, 이미 적용된 일회성 스킬은 제거

### D. 대화 히스토리 — 가장 큰 누적 요인

세션이 길어질수록 매 턴마다 이전 모든 대화가 재전송된다. 이게 가장 느려지는 직접 원인.

→ 다음 섹션에서 자세히.

## 3. 세션 운영 전략

### A. /clear — 새 세션처럼 컨텍스트 비우기

- 현재 세션의 대화 기록을 모두 비움
- CLAUDE.md·메모리·도구 정의는 유지 (시스템 프롬프트 보존)
- **작업 단위가 바뀌었을 때 즉시 호출**
- 예: 코드 리뷰 끝 → /clear → 새 기능 구현 시작

### B. /compact — 대화를 요약으로 압축

- 현재까지의 대화를 모델이 요약 → 그 요약본으로 컨텍스트 대체
- 같은 작업의 맥락은 유지하면서 토큰 절감
- 긴 디버깅 끝나고 결과만 들고 다음 단계로 넘어갈 때 유용

> **/clear vs /compact 결정 기준**:
> - 작업 자체가 끝났다 → **/clear** (가장 가볍게 시작)
> - 작업은 진행 중이지만 히스토리가 너무 길다 → **/compact** (맥락 보존)

### C. 짧은 세션 + 자주 종료

- 한 세션에 모든 걸 몰아넣지 않기
- 작업 1개 = 세션 1개 정도가 이상적
- 다음 세션은 새로 시작 → 캐시 재구성 비용은 한 번이지만 그 후 가벼움

### D. Prompt Cache 활용

- Anthropic API의 프롬프트 캐시는 **5분 TTL**
- 5분 안에 다시 호출하면 시스템 프롬프트·CLAUDE.md 부분이 캐시 히트 → 비용·속도 절감
- 5분 이상 idle하면 캐시 미스 → 다시 풀 인코딩
- → 작업 중엔 5분 안에 다음 턴을 던지는 게 비용·속도 양쪽으로 유리

## 4. 모델 선택

| 모델 | 특징 | 권장 사용처 |
|------|------|------------|
| **Opus 4.7** (`claude-opus-4-7`) | 가장 강력, 가장 느림 | 복잡한 추론·아키텍처 결정·디버깅 |
| **Sonnet 4.6** (`claude-sonnet-4-6`) | 균형잡힘 | 일반 코딩·리팩토링·문서 작성 (대부분의 일상 작업) |
| **Haiku 4.5** (`claude-haiku-4-5-20251001`) | 가장 빠름 | 단순 조회·요약·반복 작업 |

### Opus 4.6 + /fast 모드

- Opus 4.6 한정으로 `/fast` 명령 사용 가능
- 더 작은 모델로 다운그레이드되는 게 아니라, **Opus 4.6 자체에서 빠른 출력 모드** 활성화
- 추론 품질은 비슷하면서 응답 지연 감소
- Opus 4.7에선 사용 불가

### 작업별 권장
- **에이전트 호출**: 보조 작업이라면 Sonnet 또는 Haiku로 명시 (`subagent_type` 호출 시 model 파라미터)
- **메인 세션**: 작업 난이도에 맞춰 선택
- 모델 변경: `/model` 슬래시 명령 또는 `/config`

## 5. 도구 사용 최적화

### A. 병렬 도구 호출

같은 메시지 안에서 의존성 없는 도구는 한 번에 모두 호출:

```
❌ 느림: Read A → 결과 보고 → Read B → Read C
✅ 빠름: 한 메시지에 Read A, Read B, Read C 동시 호출
```

여러 파일 점검·여러 URL 조회·복수 grep을 한 번에 묶으면 응답 시간이 크게 줄어든다.

### B. Subagent로 메인 컨텍스트 보호

- 메인 세션에서 100개 파일을 grep하면 결과가 모두 메인 컨텍스트에 누적됨
- Subagent(Explore 등)에게 위임하면 결과 요약만 반환 → 메인은 가벼운 채로 유지
- 다음 턴들이 모두 빨라짐

### C. 권한 prompt 줄이기

자주 쓰는 명령이 매번 권한 prompt를 띄우면 그 자체가 시간 소모:

- 글로벌 `~/.claude/settings.json`에 자주 쓰는 도구 자동 허용 등록
- `/fewer-permission-prompts` 스킬: 최근 트랜스크립트를 분석해 자동 허용 후보를 제안
- `Bash(git status)`, `Bash(ls)`, `Read` 등 안전한 read-only는 글로벌 허용 권장

### D. Deferred tools 활용

- 시스템에 등록된 도구가 모두 매번 시스템 프롬프트에 실리면 부담
- 자주 쓰지 않는 도구는 deferred로 두고, 필요 시 `ToolSearch`로 로드
- 이건 호스트 환경에 따라 자동 관리되므로 직접 손볼 일은 적지만, 도구 카탈로그 비대를 의식하기

## 6. 정책 파일 정리

### A. settings.json 슬리밍
- 안 쓰는 권한 규칙, 만료된 hooks, 비활성 환경변수 정리
- 글로벌과 프로젝트의 중복 항목 제거 (프로젝트가 글로벌을 오버라이드하니 같은 값 중복은 무의미)

### B. 훅(hooks) 무게 점검
- Stop 훅, PostToolUse 훅이 매 턴마다 무거운 작업을 한다면 그게 응답 지연으로 직결
- 훅 스크립트 안에서 `sleep`, 무거운 grep, 외부 API 호출 등 점검
- 가능하면 훅은 **즉시 끝나는 가벼운 로직**, 무거운 작업은 비동기 큐로

본 위키 사례:
- 글로벌 Stop 훅(`log_work_history.sh`)이 매 세션 종료마다 실행됨
- 위키 폴더에서는 정책에 따라 즉시 스킵되도록 분기 → 무게 최소화

### C. MCP 서버 정리
- 안 쓰는 MCP 서버 비활성화 또는 제거
- 도구 카탈로그가 작아지면 모델의 도구 선택도 빨라지고, 시스템 프롬프트도 가벼워짐

## 7. 작업 환경·인프라

### A. 작업 폴더 위치
- iCloud Drive·Dropbox 같은 클라우드 동기화 폴더는 파일 I/O가 느릴 수 있음
- 빈번한 파일 편집·grep이 일어나는 프로젝트는 **로컬 SSD**(예: `~/Documents`, `~/Code`)에 두는 게 빠름
- 본 위키처럼 PKM이 주 목적이라면 iCloud OK, 활발한 코드 작업이면 로컬

### B. 디스크·메모리
- 시스템 자체 리소스 (디스크 90% 이상, 메모리 압박)도 응답에 영향
- `df -h`, Activity Monitor 점검

### C. 네트워크
- API 응답은 네트워크 지연에 직접 영향
- VPN·프록시 경유 시 추가 지연 가능

## 8. 실전 체크리스트 — "느려졌다" 싶을 때

```
1. ☐ 현재 세션이 너무 길지 않은가? → /clear 또는 /compact
2. ☐ CLAUDE.md 합계 줄 수 200줄 초과? → 다이어트
3. ☐ MEMORY.md 인덱스 200줄 초과? → 정리
4. ☐ 안 쓰는 agents·skills 카탈로그에 누적? → 삭제
5. ☐ 매 턴마다 권한 prompt 5개 이상? → settings.json 자동 허용 등록
6. ☐ 작업 난이도 대비 Opus 쓰고 있나? → Sonnet으로 다운
7. ☐ 도구를 직렬로 호출 중? → 병렬 호출로 묶기
8. ☐ 100개 파일 탐색을 메인에서 직접 중? → Explore subagent 위임
9. ☐ Stop·PostToolUse 훅이 무거운가? → 스크립트 점검
10. ☐ 작업 폴더가 iCloud에 있는데 빈번한 IO? → 로컬로 이동 검토
11. ☐ 5분 idle 후 재개로 캐시 미스 반복? → 작업 흐름 끊지 않기
12. ☐ MCP 서버를 10개 이상 활성화? → 안 쓰는 것 비활성
```

각 항목 5초씩 점검만 해도 즉시 효과 보는 경우가 대부분.

## 9. 안티패턴 — 흔한 함정

| 안티패턴 | 문제 | 해결 |
|----------|------|------|
| "한 세션에 모든 걸 몰아넣기" | 히스토리 누적 → 매 턴마다 토큰 폭증 | 작업 단위 세션 분리 |
| "/clear 한 번도 안 쓰기" | 끝난 작업 맥락이 영원히 컨텍스트에 머무름 | 작업 종료 직후 /clear 습관화 |
| "CLAUDE.md에 일회성 메모 적기" | 영구 누적 → 다이어트 어려워짐 | 일회성은 메모리 또는 별도 노트, CLAUDE.md엔 불변 정책만 |
| "Opus를 모든 작업에 사용" | 단순 작업도 가장 비싼 모델 | Sonnet/Haiku로 작업별 분기 |
| "권한 prompt 매번 수동 승인" | 같은 명령에 N번 prompt | 안전한 read-only는 글로벌 자동 허용 |
| "안 쓰는 에이전트·스킬 보관" | 카탈로그 비대화 | 사용하지 않으면 즉시 삭제 |
| "훅에 무거운 외부 호출" | 매 턴 지연 | 훅은 가벼운 로깅만, 무거운 건 비동기 |

## 10. 정리

- 느려지는 가장 큰 원인은 **컨텍스트 비대화**. 다이어트가 가장 효과적
- 두 번째는 **세션 길이**. /clear·/compact 적극 활용 + 짧은 세션 운영
- 세 번째는 **모델 선택**. 작업 난이도에 맞춰 Sonnet 기본, Haiku는 단순, Opus는 복잡
- 네 번째는 **인프라**. 권한 prompt·훅·파일 위치까지 점검

> **30초 룰**: "느려졌다" 싶으면 일단 `/clear` 한 번. 그래도 느리면 위 체크리스트 1번부터 12번까지 5초씩 점검.
