# AI 정책파일 관리 — 글로벌 vs 프로젝트 용도 구분

> 작성일: 2026-05-08
> 출처: Claude Code 공식 문서 (https://code.claude.com/docs) + 본 위키의 실제 운영 사례

Claude Code를 오래 쓰다 보면 `CLAUDE.md`, `settings.json`, `.claude/agents/`, `.claude/skills/`, 메모리 파일 등 정책 파일이 빠르게 늘어난다. **어떤 정책을 글로벌(`~/.claude/`)에 두고, 어떤 것을 프로젝트(`<repo>/.claude/`) 안에 둘지** 잘못 판단하면, 다른 프로젝트로 넘어갔을 때 엉뚱한 규칙이 적용되거나 같은 정책을 여러 곳에서 중복 관리하는 사태가 생긴다. 이 글은 정책 파일별 위계·우선순위·용도 구분 기준을 정리한다.

---

## 1. 정책 파일 위계 한눈에

| 정책 파일 | 글로벌 위치 | 프로젝트 위치 | 머징/우선순위 |
|----------|------------|--------------|---------------|
| **CLAUDE.md** | `~/.claude/CLAUDE.md` | `<repo>/CLAUDE.md` | 둘 다 로드, **연결됨** (글로벌 → 프로젝트 순) |
| **settings.json** | `~/.claude/settings.json` | `<repo>/.claude/settings.json` | 프로젝트가 글로벌을 **오버라이드** |
| **settings.local.json** | (없음, 비추천) | `<repo>/.claude/settings.local.json` | 프로젝트 settings를 다시 오버라이드 (개인 로컬, gitignore 권장) |
| **agents** | `~/.claude/agents/*.md` | `<repo>/.claude/agents/*.md` | 둘 다 사용 가능, **이름 충돌 시 프로젝트 우선** |
| **skills** | `~/.claude/skills/<name>/SKILL.md` | `<repo>/.claude/skills/<name>/SKILL.md` | 둘 다 사용 가능, **이름 충돌 시 프로젝트 우선** |
| **memory** | (글로벌 메모리 별도) | 프로젝트 디렉토리 해시 기준 격리 | 자동으로 분리 — 신경 X |

> **핵심 원칙**: `CLAUDE.md`는 **누적**(둘 다 적용), 그 외 대부분은 **오버라이드**(프로젝트가 우선).

---

## 2. CLAUDE.md — 가장 자주 쓰는 정책 파일

### 머징 순서

Claude Code는 세션 시작 시 다음 순서로 모든 `CLAUDE.md`를 컨텍스트에 로드한다:

1. **글로벌**: `~/.claude/CLAUDE.md`
2. **프로젝트 루트**: `<cwd 상위 git 루트>/CLAUDE.md`
3. **현재 작업 디렉토리에 가까운 서브디렉토리 CLAUDE.md** (예: `<repo>/frontend/CLAUDE.md`)
4. **`@import` 임포트된 외부 파일**

→ 모두 누적된다. 충돌하는 지시가 있으면 **나중에 로드된(더 가까운) 것이 우선**으로 해석되는 경향이 있다.

### 어떤 내용을 어디에?

#### ✅ 글로벌(`~/.claude/CLAUDE.md`)에 두기 좋은 것
- **개인 정체성**: 이메일, 역할, 일반적 작업 스타일
- **모든 프로젝트에 공통되는 도구 선호**: 자주 쓰는 슬래시 명령, 응답 톤, 코드 스타일 일반론
- **개인 자동화 시스템**: 주간보고 훅, 메모리 정책 같은 글로벌 인프라
- **여러 프로젝트를 가리키는 포인터**: "MyClaude 앱은 `~/Documents/Code/MyClaude`에 있음"

#### ✅ 프로젝트(`<repo>/CLAUDE.md`)에 두기 좋은 것
- **그 프로젝트만의 컨벤션**: 파일명 규칙, 폴더 구조, 도메인 용어
- **도메인 지식**: 비즈니스 규칙, 외부 시스템 연동 정보
- **빌드·배포·테스트 명령**: `npm run dev`, `bun test`, `./start.sh` 등
- **자동화 트리거**: "이런 발화 시 이런 워크플로우" 같은 프로젝트 특화 규칙
- **글로벌 정책의 예외 선언**: "이 프로젝트에선 글로벌 X 규칙 적용 안 함"

### 위치별 우선순위 활용 — 글로벌 정책 무력화

이 위키의 실제 사례:

> **글로벌** `~/.claude/CLAUDE.md`에는 "주간보고용 작업 히스토리" 자동 기록 시스템이 정의되어 있다.
> **위키 프로젝트** `wiki/CLAUDE.md`에는 "주간보고 작업 히스토리 제외" 섹션이 추가되어 있다.

→ 위키 폴더에서 작업할 때만 글로벌 자동 기록을 끈다. 글로벌 훅 자체는 그대로 두고, 그 적용 범위를 프로젝트 단위로 좁혔다. **글로벌 인프라 + 프로젝트 단위 예외**라는 패턴이 깔끔하다.

## 3. settings.json — 권한·환경변수·훅

### 위계
- 글로벌: `~/.claude/settings.json`
- 프로젝트: `<repo>/.claude/settings.json`
- 로컬 개인: `<repo>/.claude/settings.local.json` (gitignore에 추가 권장)

### 머징
**오버라이드 방식**. 프로젝트 settings가 같은 키를 가지면 글로벌 값을 덮어쓴다.

### 어디에 둘 것인가

#### ✅ 글로벌
- 자주 쓰는 도구의 자동 허용 (`Read`, `Bash(git status)`, `Bash(ls)`)
- 글로벌 hooks (예: 주간보고 Stop 훅)
- 모든 프로젝트에 공통되는 환경변수

#### ✅ 프로젝트
- 그 프로젝트의 특수 명령 허용 (`Bash(npm run *)`, `Bash(bq *)`)
- 프로젝트 전용 환경변수
- 프로젝트 특화 훅 (예: 특정 디렉토리에 commit 시 lint 실행)

#### ✅ 로컬 (`.local.json`)
- 개인 토큰·시크릿 (절대 커밋하지 않음)
- 본인만 쓰는 일시 권한·디버그 설정

## 4. Agents — 프로젝트 전속 페르소나

### 위치
- 글로벌: `~/.claude/agents/<agent-name>.md`
- 프로젝트: `<repo>/.claude/agents/<agent-name>.md`

이름 충돌 시 프로젝트가 우선.

### 어디에 둘 것인가

#### ✅ 글로벌
- 모든 프로젝트에서 쓰고 싶은 일반 페르소나 (예: 일반 코드 리뷰어, 보안 감사자)
- 도메인 무관한 보조 역할

#### ✅ 프로젝트
- 프로젝트 도메인에 깊게 의존하는 페르소나
- 프로젝트 데이터 파일을 읽어 일관성을 유지하는 페르소나

**본 위키의 실제 사례**:
| 에이전트 | 위치 | 이유 |
|----------|------|------|
| `golf-coach` (한지민) | `wiki/.claude/agents/` | 위키의 `golf/coaching-log.md`를 참조해야 하므로 위키에 종속 |
| `warren-buffett` (버핏형) | `wiki/.claude/agents/` | 위키의 `stock/_포트폴리오.md`를 읽고 갱신해야 하므로 위키에 종속 |

→ 둘 다 **데이터 파일과 함께 있어야** 의미가 있어서 프로젝트 위치가 정답.

## 5. Skills — 절차·체크리스트

### 위치
- 글로벌: `~/.claude/skills/<name>/SKILL.md`
- 프로젝트: `<repo>/.claude/skills/<name>/SKILL.md`

### 어디에 둘 것인가

#### ✅ 글로벌
- 어디서든 쓸 수 있는 일반 절차 (`/commit`, `/code-review-checklist`, `/security-review`)
- 도구 사용법 가이드 (Claude API, 키바인딩 설정 같은 메타 스킬)

#### ✅ 프로젝트
- 프로젝트 도메인 절차 (`/golf-reservation`, `/deploy-staging`)
- 프로젝트 파일 조작 자동화

**본 위키의 사례**:
- `wiki/.claude/skills/golf-reservation/` — 골프 예약 처리 (daily 로그 + macOS 캘린더 연동). 위키 데이터에 종속이라 프로젝트 위치 정답.

## 6. 메모리 — 자동 격리

Claude Code의 자동 메모리 시스템은 **프로젝트 디렉토리 해시 기준으로 격리**된다. 사용자가 신경 쓸 것 없이:

- 위키 폴더에서 쌓은 메모리는 **위키 작업할 때만** 로드
- 다른 프로젝트에서 쌓은 메모리는 그 프로젝트에서만

→ "글로벌 vs 프로젝트" 고민이 거의 필요 없는 영역. 단, 명시적으로 "이건 모든 작업에 공통" 같은 사실은 글로벌 메모리(있다면)에 두는 게 맞다.

## 7. 의사결정 기준 — 어디에 둘 것인가?

```
정책을 작성할 때 자문:
│
├─ "이건 모든 프로젝트에서 공통될까?"
│   ├─ 예 → 글로벌
│   └─ 아니오 → 프로젝트
│
├─ "이건 특정 프로젝트의 데이터·도메인에 의존할까?"
│   ├─ 예 → 프로젝트
│   └─ 아니오 → 글로벌
│
├─ "이건 시크릿이거나 본인만 쓰는 설정일까?"
│   ├─ 예 → 프로젝트의 `.local.json`
│   └─ 아니오 → 글로벌 또는 프로젝트
│
└─ "글로벌에 두긴 했는데 특정 프로젝트에서만 끄고 싶다"
    └─ 글로벌은 그대로 두고, **프로젝트 CLAUDE.md에 예외 선언**
```

## 8. 안티패턴 — 흔한 실수

| 안티패턴 | 문제 | 해결 |
|----------|------|------|
| 모든 정책을 글로벌에 몰아둠 | 다른 프로젝트로 옮길 때 잘못된 컨벤션 적용, 도메인 노출 | 도메인 의존 정책은 프로젝트로 이동 |
| 모든 정책을 프로젝트에 복사 | 같은 규칙을 N번 관리, 동기화 안 됨 | 공통 규칙은 글로벌, 차이점만 프로젝트 |
| 글로벌 시크릿을 settings.json에 | 다른 프로젝트에서 의도치 않게 노출 | `.local.json`로 이동 + gitignore |
| 프로젝트 CLAUDE.md에 개인 식별자 적기 | 협업·git 공유 시 노출 | 글로벌로 이동, 프로젝트엔 일반 컨벤션만 |
| 프로젝트 정책을 글로벌에 적고 "특정 폴더에서만"이라 작성 | 다른 폴더에서 의도치 않게 발동 | 프로젝트 CLAUDE.md로 이동 또는 디렉토리 가드 명시 |
| 같은 에이전트를 글로벌·프로젝트 양쪽에 정의 | 어느 쪽이 발동되는지 혼란 | 한쪽으로 통일, 충돌 회피 |

## 9. 실전 예시 — 본 위키의 정책 분포

### `~/.claude/` (글로벌)
- `CLAUDE.md`: 위키·MyClaude 앱 폴더 위치 포인터, 주간보고 자동 기록 시스템 (모든 프로젝트 적용)
- `scripts/log_work_history.sh`: Stop 훅 스크립트
- `settings.json`: Stop 훅 등록, 글로벌 도구 권한
- `work_history/`: 주간보고 누적 디렉토리

### `wiki/` (프로젝트)
- `CLAUDE.md`: 위키 컨벤션 (Daily 규칙, 골프, 주식 분석, 캘린더, 주차빌런, 아티클, Git, 디스코드 봇, **주간보고 제외 선언**)
- `.claude/agents/`: `golf-coach`, `warren-buffett` (도메인 데이터 종속)
- `.claude/skills/`: `golf-reservation`
- `discord-bot/bot.py`: 위키 자체 디스코드 봇 (이 프로젝트 전용 자동화)

### 정책 분배의 논리
- 주간보고 시스템은 **모든 프로젝트에 공통이라 글로벌**
- 위키만 예외(개인 PKM이라 보고 대상 아님)이므로 **위키 CLAUDE.md에 예외 선언**
- 골프·주식 에이전트는 **위키 데이터 없이는 무의미하므로 프로젝트 종속**

## 10. 정리

- `CLAUDE.md`는 **글로벌 + 프로젝트 누적**, 다른 정책 파일은 **프로젝트가 글로벌 오버라이드**
- 도메인·데이터에 의존하면 **프로젝트**, 무관하게 모든 작업에 적용되면 **글로벌**
- 시크릿·개인 설정은 **`.local.json`** (gitignore)
- 글로벌 정책을 특정 프로젝트에서만 끄려면 **프로젝트 CLAUDE.md에 예외 선언**
- 같은 정책을 양쪽에 복사하지 말 것 — 공통은 한쪽, 차이점만 프로젝트로

이 원칙만 지키면 정책 파일 30~50개로 늘어나도 어디에 무엇이 있는지 헷갈리지 않는다.
