# Claude Code의 Skills, Subagents, Teams 완벽 가이드

> 작성일: 2026-05-08
> 출처: Claude Code 공식 문서 (https://code.claude.com/docs)

Claude Code는 세 가지 핵심 기능으로 AI 어시스턴트의 능력을 확장한다: **Skills(스킬)**, **Subagents(서브에이전트)**, **Teams(팀)**. 겉으로는 비슷해 보이지만 각각 다른 문제를 풀기 위해 설계되었다. 이 글은 셋의 정의·차이·선택 기준을 학습용으로 정리한다.

---

## 1. 각 개념의 정의

### Skill

**Skill**은 특정 작업을 위한 재사용 가능한 **지시문 꾸러미**다. 체크리스트, 템플릿, 절차서처럼 동작한다.

**파일 위치**
- 개인: `~/.claude/skills/<skill-name>/SKILL.md`
- 프로젝트: `.claude/skills/<skill-name>/SKILL.md`
- 플러그인: `<plugin>/skills/<skill-name>/SKILL.md`

**SKILL.md 예시**
```markdown
---
name: my-skill
description: 이 스킬이 무엇을 하는지, 언제 쓰일지 (자동 호출 판단 기준)
allowed-tools: Read, Bash(git *)
---

# 스킬 본문 (지시문/절차/참고자료)

1. 먼저 git status로 변경 내용 확인
2. ...
```

**컨텍스트 특징**
- Skill은 **현재 세션 컨텍스트 내에서** 실행된다
- 호출 시 본문이 메인 대화에 로드되어 토큰을 소모하고 그 이후에도 유지됨
- 따라서 **짧고 재사용성 높은 지시문**에 적합

**호출 방식**
- 명시적: `/skill-name` 슬래시 명령
- 자동: frontmatter `description`을 보고 Claude가 매칭해 호출 판단

---

### Subagent

**Subagent**는 **독립된 컨텍스트 윈도우를 가진 전문 AI 어시스턴트**다. 메인 세션과 별도의 메모리 공간에서 일하고, 결과만 메인으로 돌려준다.

**파일 위치**
- 개인: `~/.claude/agents/<agent-name>.md`
- 프로젝트: `.claude/agents/<agent-name>.md`

**에이전트 정의 예시** (실제 `golf-coach.md`)
```markdown
---
name: golf-coach
description: 사용자의 전속 골프 코치 '한지민(지민누나)'. 사용자가 "지민누나"라고
  호칭하거나, 골프 스윙·자세·구질·드릴·라운드 피드백 조언을 구할 때 자동 호출.
tools: Read, Write, Edit, Grep, Glob
---

당신은 사용자의 전속 골프 코치 한지민입니다.
지난 코칭 로그를 읽고 일관성 있는 피드백을 제공하세요...
```

> **주의**: 자동 호출 트리거는 별도 `when_to_invoke` 필드가 아니라 **`description` 안에 자연어로** 적는다. Claude가 사용자 발화와 description을 매칭해 자동 호출 여부를 판단한다.

**컨텍스트 특징**
- 메인 세션 대화 이력을 **공유받지 않음** → 매번 자기소개급 컨텍스트 필요
- 복잡한 파일 탐색·검색·연구를 해도 **메인 대화는 오염되지 않음**
- 작업 완료 후 **요약된 결과 한 번**만 메인으로 반환

**호출 방식**
- 명시적: 사용자가 직접 호출 (`@<agent-name>` 또는 슬래시 명령, 환경에 따라)
- 자동: description 매칭으로 Claude가 자동 호출

---

### Team

**Team**은 **여러 Claude Code 세션을 병렬로 조율하는 구조**다. 한 세션이 리더(lead)가 되고, 여러 팀원(teammate)이 각자 다른 작업을 동시에 수행한다.

**구조**
- **Team Lead**: 작업을 분해·할당·취합하는 주 세션
- **Teammates**: 각자 독립된 세션·컨텍스트에서 동시에 일함
- **Shared Task List**: 모든 팀원이 보는 공통 작업 목록
- **Messaging**: 팀원 간 **직접 메시지** 송수신 가능 (Subagent는 리더에게만 보고하지만, Team 팀원은 서로 대화)

**컨텍스트 특징**
- 각 팀원이 **독자적인 컨텍스트 윈도우**를 가짐
- 리더 대화 이력은 팀원에게 자동 전파되지 않음 → 작업 지시 시 명시적으로 전달
- 팀원 간 협업이 가능하다는 점이 Subagent와 결정적으로 다름

**호출 방식**
- 자연어 요청: "3명의 팀을 만들어 보안·성능·테스트 관점으로 동시 분석해줘"
- Claude가 자동으로 팀 구성 또는 사용자가 명시적 역할 지정
- 활성화 방법은 사용 환경에 따라 다르므로 [공식 문서](https://code.claude.com/docs)를 참고

## 2. 언제 써야 하는가 (Use Cases)

### Skill을 써야 할 때

1. **반복 체크리스트/절차**: "커밋 전 항상 확인할 6가지"
2. **참고 자료/규칙 모음**: API 컨벤션, 코딩 규칙, 보안 체크
3. **간단한 자동화 명령**: `/deploy`, `/test-all` 같은 단축키
4. **5~10분 안에 끝나는 작은 흐름**

예시: `/commit`, `/code-review-checklist`, `/api-standards`

---

### Subagent를 써야 할 때

1. **메인 컨텍스트를 보호하고 싶은 부수적 조사**: 라이브러리 문서를 통째로 읽고 요약해서 가져오기
2. **고정된 역할/페르소나가 반복**: 골프 코치, 보안 검토자, 코드 설명가
3. **다양한 도구 사용이 예상되지만 메인 대화는 깔끔하게**: 수많은 파일 grep·read를 시켜도 메인은 영향 없음
4. **결과만 필요하고 과정은 불필요**

예시: `golf-coach`(코칭 로그 누적), `security-reviewer`(PR 보안 검토), `warren-buffett`(포트폴리오 브리핑)

---

### Team을 써야 할 때

1. **대규모 병렬 작업**: 여러 모듈을 동시에 리팩토링
2. **여러 관점 동시 탐색**: 보안·성능·테스트를 각각 다른 팀원이
3. **상호 검증·논박이 필요한 문제**: 가설 5개를 팀원에게 분배해 서로 반박하며 진짜 원인 찾기
4. **단일 Subagent의 직렬 처리로는 답답한 복잡한 문제**

예시: 병렬 코드 리뷰, 다중 가설 디버깅, 대규모 기능 동시 개발

---

## 3. 결정 트리

```
"이 작업을 어떻게 처리할까?"
│
├─ 지시문/절차/참고자료다  ───────────→ SKILL
│
├─ 특정 인물/역할이 반복된다 ──────────→ SUBAGENT
│   (코치, 검토자, 설명가, 분석가...)
│
├─ 여러 명이 병렬로 일하고 협력해야 한다 → TEAM
│
└─ 위 셋 다 아니다 ────────────────────→ 그냥 메인 Claude에게 직접 시키기
```

## 4. 비교 표

| 항목 | **Skill** | **Subagent** | **Team** |
|------|-----------|--------------|----------|
| **정의 위치** | `.claude/skills/<name>/SKILL.md` | `.claude/agents/<name>.md` | 자연어로 동적 구성 |
| **실행 위치** | 현재 세션 내 | 독립 세션 (단일) | 다중 독립 세션 (리더 + 팀원 N) |
| **컨텍스트 격리** | ❌ 메인에 흡수 | ✅ 독립 | ✅ 팀원별 독립 |
| **메인 대화 오염** | 높음 (본문이 컨텍스트에 적재) | 낮음 (결과만 반환) | 낮음 (결과만 반환) |
| **통신 패턴** | (해당 없음) | 리더에게만 결과 보고 | **팀원끼리 직접 메시지** |
| **호출 방식** | `/skill-name` 또는 자동 | 명시적/자동(description 매칭) | 자연어로 팀 구성 요청 |
| **병렬성** | 없음 | 단일 세션 | **다중 세션 병렬** |
| **토큰 비용** | 낮음 | 중간 | **높음** (세션 수 × 비용) |
| **대표 용도** | 절차·참고자료·짧은 자동화 | 고정 페르소나·격리 조사 | 다관점 병렬 분석·협력 |

## 5. 실전 시나리오

### 시나리오 1: GitHub PR 검토

**상황**: 큰 PR이 들어왔다. 보안·성능·테스트 커버리지를 모두 점검해야 한다.

| 방식 | 결과 |
|------|------|
| **Skill** | ⚠️ 비효율 — 한 번에 모든 관점을 직렬로 훑게 됨. 한 영역에 집중하다 다른 걸 놓치기 쉬움. |
| **Subagent** | ⚠️ 결과만 리더에 보고되어, 보안 관점 발견 사항을 성능 관점이 곧장 받아 추격 검증하기 어려움. |
| **Team** | ✅ 최적 — 보안·성능·테스트 3명이 동시에 분석하고, 서로 메시지로 "이 보안 이슈가 성능 핫패스에도 있어?"라고 협력 가능. |

---

### 시나리오 2: 버그 근본 원인 찾기

**상황**: "메시지 1개 보낸 뒤 연결이 끊긴다." 원인은? 가설이 너무 많다.

| 방식 | 결과 |
|------|------|
| **Skill** | ❌ 부적합 — 절차만 제시하고 실제 원인 추적은 못함. |
| **Subagent** | ⚠️ "타임아웃·메모리·세션·네트워크·동시성" 5가지를 리더가 다시 하나하나 검증해야 함. 직렬·느림. |
| **Team** | ✅ 최적 — 5명 팀원이 각자 다른 가설(타임아웃/메모리/세션/네트워크/동시성)로 동시 조사. 팀원 간 "그 가설은 X 로그가 반증해" 식 논박. 가장 강한 증거를 가진 가설이 우승. |

## 6. 빠른 선택 가이드

| 상황 | 답 | 이유 |
|------|----|------|
| 항상 쓰는 체크리스트 | **Skill** | 재사용성, 낮은 비용 |
| "내 골프 코치처럼 조언해줘" | **Subagent** | 고정 역할, 독립 컨텍스트 |
| 다관점 병렬 동시 분석 | **Team** | 병렬성 + 상호 협력 |
| 일회성 단순 질문 | **그냥 메인 Claude** | 오버엔지니어링 회피 |
