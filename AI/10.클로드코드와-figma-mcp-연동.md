> 출처: Figma Dev Mode MCP Server 공식 문서 (https://help.figma.com/hc/en-us/articles/32132100833559) + Claude Code MCP 문서 (https://docs.claude.com/en/docs/claude-code/mcp) + 본 위키 실전 경험
> 형식: Figma 디자인을 Claude Code로 실제 앱 UI 코드로 만드는 실전 가이드

UI를 짜는 가장 빠른 방법은 더 이상 "디자이너가 시안 던지면 개발자가 다시 짜기"가 아니다. **Figma에 있는 디자인을 Claude Code가 직접 읽고, 디자인 시스템에 맞춰 코드로 뽑게** 하는 흐름이 가능해졌다. 핵심은 Figma의 공식 **Dev Mode MCP Server**.

이 글은 그 연결을 어떻게 셋업하고, 어떻게 디자인 시스템을 학습시키고, 실전에서 어떻게 지시해야 결과물이 안 깨지는지를 한 번에 정리한다.

---

## 0. 왜 Figma MCP인가 — 스크린샷 붙여넣기와의 차이

이전엔 디자인을 코드로 옮기는 방법이 두 가지였다.

| 방식 | 정확도 | 문제 |
|------|--------|------|
| 스크린샷 → AI 코딩 | 낮음 | 토큰·간격·폰트 다 추정. 디자인 시스템 무시 |
| 디자이너 spec 문서 → 수동 코딩 | 높음 | 느림. 디자인 변경 시 매번 다시 |
| **Figma MCP → Claude Code** | **높음+자동** | 변수·컴포넌트·토큰을 그대로 읽고 코드 생성 |

Figma MCP는 단순히 "이미지를 본다"가 아니다. **노드의 변수(variable), 컴포넌트 인스턴스, Auto Layout 속성, Code Connect 매핑**까지 그대로 가져와서 코드로 변환한다. 토큰을 추정할 일이 없어진다.

## 1. 기본 세팅 — Figma Dev Mode MCP Server 켜기

### 전제 조건
- **Figma Desktop App** (브라우저로는 불가) 최신 버전
- **Professional / Organization / Enterprise 플랜** (Starter 불가, Dev Mode 권한 필요)
- Claude Code 최신 버전

### Figma 쪽 활성화
1. Figma Desktop 앱 열기
2. 왼쪽 상단 메뉴 → **Preferences** → **Enable Dev Mode MCP Server** 체크
3. 활성화되면 로컬에 SSE 엔드포인트가 뜬다: `http://127.0.0.1:3845/sse`

> Figma 앱이 실행 중일 때만 MCP 서버가 살아있다. 앱을 닫으면 끊긴다.

### Claude Code 쪽 등록

**옵션 A — CLI로 등록 (가장 단순)**
```bash
claude mcp add --transport sse figma http://127.0.0.1:3845/sse
```

**옵션 B — `.mcp.json`에 직접 (팀 공유용)**
프로젝트 루트의 `.mcp.json`:
```json
{
  "mcpServers": {
    "figma": {
      "type": "sse",
      "url": "http://127.0.0.1:3845/sse"
    }
  }
}
```

> 팀 레포라면 옵션 B. 모든 팀원이 같은 MCP 서버 정의를 공유한다. Figma 앱은 각자 켜야 함.

### 연결 확인
Claude Code에서:
```
/mcp
```
`figma` 항목이 connected로 뜨면 성공. 사용 가능한 도구는 보통 다음과 같다.

| 도구 | 역할 |
|------|------|
| `get_code` | 선택된 노드를 코드로 (기본 React + Tailwind) |
| `get_variable_defs` | 변수·색·토큰 정의 |
| `get_code_connect_map` | Figma 컴포넌트 ↔ 실제 코드 컴포넌트 매핑 |
| `get_image` | 디자인 스크린샷 |
| `create_design_system_rules` | 디자인 시스템 규칙 생성 |

## 2. UI/UX 디자인 에이전트 만들기

기본 세팅이 끝나면, 그냥 Claude Code에게 "이거 코드로 만들어줘"라고 해도 동작은 한다. 하지만 **매번 같은 품질을 뽑으려면 전용 서브에이전트로 분리**하는 게 답이다.

### 왜 에이전트로 분리하나
- 디자인 작업은 **컨텍스트가 거대**하다 (디자인 시스템, 컴포넌트 라이브러리, 토큰 규칙)
- 본 세션 컨텍스트를 오염시키지 않고 격리된 컨텍스트에서 처리
- 일관된 코딩 컨벤션 강제 (Tailwind vs CSS Modules vs styled-components 등)

### 에이전트 파일 (`.claude/agents/ui-designer.md`)

```markdown
---
name: ui-designer
description: Figma 디자인을 실제 앱 UI 코드(React + Tailwind)로 변환하는 전문가. "이 화면 만들어줘", "Figma 링크" 같은 요청 시 자동 호출.
tools: [Read, Write, Edit, Bash, Grep, Glob, mcp__figma__get_code, mcp__figma__get_variable_defs, mcp__figma__get_code_connect_map, mcp__figma__get_image]
---

너는 시니어 프론트엔드 엔지니어다. Figma MCP를 통해 디자인을 읽고,
프로젝트의 디자인 시스템 규칙을 100% 준수하는 코드를 생산한다.

## 절대 규칙
1. 색·간격·폰트 사이즈는 **반드시 디자인 토큰(`src/styles/tokens.ts`)
   참조**. 하드코딩된 hex/px 금지.
2. 컴포넌트는 우선 **`src/components/ui/`의 기존 컴포넌트 재사용 검토**
   후 신규 생성.
3. Figma의 Auto Layout은 **flex/gap으로**, Constraint는 **absolute 회피**.
4. 한 화면 = 한 파일 ❌. **섹션 단위로 분리**해서 작성.

## 작업 흐름
1. `get_variable_defs`로 사용된 변수 먼저 확인 → 토큰 매핑 확정
2. `get_code_connect_map`으로 기존 컴포넌트 매핑 확인
3. `get_code`로 코드 초안 받기
4. 위 규칙대로 정제·재작성
5. 결과는 항상 **타입 안전 (TypeScript) + 접근성 속성(aria-*) 포함**

## 절대 하지 않을 것
- `get_image`만 보고 추정하기 (마지막 보루)
- 디자인 시스템에 없는 색·간격 임의 도입
- `<div>` 남발 (시맨틱 태그 우선)
```

### 호출 방식
```
@ui-designer 이 Figma 화면 코드로 만들어줘
[Figma URL 또는 Figma 앱에서 노드 선택 상태]
```

> 에이전트가 `mcp__figma__*` 도구를 갖고 있어야 본 세션을 거치지 않고 직접 MCP 호출이 가능하다. tools 배열에 반드시 명시.

## 3. 디자인 시스템 학습시키기

Figma MCP의 진짜 가치는 **디자인 시스템을 그대로 코드에 동기화**할 수 있다는 점이다. 이게 안 되면 결과물은 매번 토큰이 어긋난다.

### 1단계 — Figma 디자인 시스템 정리 (디자이너 영역)
- 모든 색은 **Color Variable**로 (Primary/500 같은 토큰명)
- 간격·폰트도 **Number Variable / Text Style**로
- 자주 쓰는 UI는 **Component**로 묶기
- 컴포넌트 변형은 **Variants**로

이게 안 되어 있으면 MCP가 뽑는 코드도 그냥 raw hex·magic number로 나온다. **MCP는 디자인 시스템을 만들어주지 않는다, 동기화만 해준다.**

### 2단계 — Figma 변수를 코드 토큰으로 추출

Claude Code에 한 번 시키자.

```
@ui-designer Figma의 색·간격·타이포 변수를 모두 가져와서
src/styles/tokens.ts 파일로 만들어줘. Tailwind config에도
연동되게 해줘.
```

에이전트가 `get_variable_defs` 호출 → 변수 트리 받음 → TS 파일로 변환.

결과 예시 (`src/styles/tokens.ts`):
```typescript
export const colors = {
  primary: {
    50: '#EBF5FF',
    500: '#3B82F6',
    900: '#1E3A8A',
  },
  // ...
} as const;

export const spacing = {
  xs: '4px',
  sm: '8px',
  md: '16px',
  // ...
} as const;
```

그리고 `tailwind.config.ts`에서 이 토큰을 참조하게 연결.

### 3단계 — Code Connect 매핑 (강력 권장)

Figma의 한 컴포넌트가 실제 코드의 어떤 컴포넌트와 1:1 대응되는지 명시하는 기능. 디자이너가 Figma에서 `<Button>` 컴포넌트를 쓰면, MCP가 자동으로 `import { Button } from '@/components/ui/button'`를 생성한다.

- Figma 측: 각 컴포넌트에 Code Connect snippet 첨부 (디자이너 협조 필요)
- Claude 측: `get_code_connect_map` 호출로 매핑 가져옴

> 매핑이 있으면 MCP가 똑같은 버튼을 매번 새로 그리지 않고 **기존 컴포넌트를 재사용**한다. 코드 일관성 측면에서 결정적 차이.

### 4단계 — CLAUDE.md에 규칙 박기

프로젝트 루트의 `CLAUDE.md`:
```markdown
## UI 작업 규칙 (Figma MCP 연동)

- 모든 색·간격·타이포는 `src/styles/tokens.ts` 참조
- 신규 컴포넌트 만들기 전 `src/components/ui/` 재사용 검토 필수
- Figma 디자인 변환 작업은 반드시 `@ui-designer` 에이전트 호출
- Figma 변수가 새로 추가되면 → `tokens.ts` + `tailwind.config.ts` 동시 갱신
```

이게 있어야 본 세션 클로드든 서브에이전트든 같은 규칙으로 움직인다.

## 4. 실전 디자인 지시하기

세팅·시스템 학습이 끝나면 이제 실제 지시. 케이스별로 정리.

### Case A — 새 화면 통째로 만들기

**Bad**: "Figma에 있는 로그인 화면 만들어줘"

**Good**:
```
@ui-designer 아래 Figma 노드를 받아서:
1. 화면 전체는 src/screens/Login/index.tsx
2. 입력 폼은 src/screens/Login/LoginForm.tsx로 분리
3. SocialLoginButton은 src/components/ui/SocialButton 재사용
4. 모든 색·간격은 토큰 사용

[Figma URL: https://figma.com/file/.../?node-id=123-456]
```

차이: **파일 구조·재사용 방향·토큰 규칙을 명시**. 이것만 있으면 결과물의 일관성이 두 배 이상.

### Case B — 부분 컴포넌트만 가져오기

Figma 앱에서 해당 노드를 **선택한 상태**로 Claude에게:
```
@ui-designer 지금 선택된 카드 컴포넌트만 ProductCard.tsx로 만들어줘.
Variants는 (default/highlighted/disabled) 3개 다 지원되게.
```

Selection-based가 URL-based보다 빠르고 정확. Figma에서 의도한 정확한 노드만 가져온다.

### Case C — 디자인 변경 반영

Figma에서 디자인이 수정됐을 때:
```
@ui-designer Figma의 ProductCard 다시 가져와서
src/components/ProductCard.tsx와 diff 봐줘.
바뀐 부분만 패치로 반영, 기존 props 시그니처는 유지.
```

핵심: **전체 재생성 금지, diff 기반 패치**. 안 그러면 직접 손본 부분이 날아간다.

### Case D — 디자인 시스템 자체가 바뀜

```
@ui-designer Figma 변수 다시 가져와서 tokens.ts 갱신해줘.
삭제된 토큰이 있다면 코드에서 사용처 grep해서 보고만 해줘 (수정은 보류).
```

토큰 변경은 영향 범위가 크다. **변경 → 영향 분석 → 수동 확인 → 수정** 흐름.

## 5. 자주 깨지는 지점 — 체크리스트

| 증상 | 원인 | 해결 |
|------|------|------|
| 색이 hex 하드코딩으로 나옴 | Figma에 Color Variable 미정의 | 디자이너에게 변수화 요청 |
| 같은 버튼이 매번 새로 그려짐 | Code Connect 매핑 없음 | Figma 컴포넌트에 매핑 추가 |
| 간격이 들쭉날쭉 | Auto Layout 미사용 | Figma 측 Auto Layout 적용 |
| `get_code` 결과가 비어있음 | 노드 미선택 / URL 잘못됨 | Selection 또는 정확한 node-id |
| MCP가 안 잡힘 | Figma Desktop 미실행 | 데스크탑 앱 켜기, `/mcp` 재확인 |
| 결과 React인데 우리 프로젝트는 Vue | get_code 기본값이 React | 에이전트 프롬프트에 프레임워크 명시 |

## 6. 권장 워크플로우 한 장 요약

```
[디자이너] Figma 디자인 시스템 정리 (변수·컴포넌트·Code Connect)
              ↓
[1회 셋업]   Figma MCP 등록 → ui-designer 에이전트 생성
              ↓
[1회 셋업]   변수 추출 → tokens.ts + tailwind.config 연동
              ↓
[반복]       @ui-designer 호출 → 파일 구조·재사용 방향 명시
              ↓
[검수]       토큰 사용 여부·접근성·재사용성 체크
              ↓
[변경 시]    diff 기반 패치 (전체 재생성 금지)
```

---

## 한 줄 요약

**Figma MCP는 디자인을 코드로 "옮기는" 도구가 아니라, 디자인 시스템을 코드와 "동기화"하는 도구다.** 시스템이 정리되어 있을수록 결과물이 깨지지 않고, 에이전트로 격리할수록 일관성이 올라간다. 셋업 한 번 잘해두면, 그 다음부턴 "Figma에서 이 화면 만들어줘" 한 줄로 끝난다.
