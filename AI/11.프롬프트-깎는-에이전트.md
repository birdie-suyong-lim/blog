# 프롬프트 깎는 에이전트 — 프롬프트를 다듬는 전용 에이전트 만들기

> 작성일: 2026-06-01
> 출처: Claude Code / Anthropic SDK 문서 + 메타프롬프팅(meta-prompting) 실무 경험

좋은 답을 받는 가장 빠른 길은 좋은 모델을 쓰는 게 아니라 **좋은 프롬프트를 쓰는 것**이다. 그런데 좋은 프롬프트를 매번 손으로 쓰는 건 피곤하다. 그래서 나오는 발상이 이거다 — "내가 대충 쓴 프롬프트를, 본 작업에 넘기기 전에 한 번 다듬어주는 에이전트를 두면 어떨까?"

이게 **프롬프트 깎는 에이전트(prompt refiner)**다. 사용자의 거친 요청을 받아 → 의도를 명료화하고 → 빠진 맥락·제약·출력 형식을 채워 → 잘 구조화된 프롬프트로 "깎아서" 본 작업 모델에게 넘긴다. 나무를 깎듯 거친 표면을 다듬는다는 의미에서 "깎는다".

이 글은 이걸 **무엇인지 → 언제 쓰는지 → 어떻게 만드는지 → 장단점**의 순서로 정리한다.

---

## 1. 개념 — 프롬프트를 깎는다는 것

핵심은 **요청과 실행 사이에 다듬는 단계를 하나 끼우는 것**이다.

```
[기존]
사용자 거친 요청 ──────────────▶ 본 작업 모델 ──▶ 결과
                              (의도 추측·맥락 부족으로 빗나감)

[프롬프트 깎는 에이전트]
사용자 거친 요청 ──▶ 깎는 에이전트 ──▶ 다듬어진 프롬프트 ──▶ 본 작업 모델 ──▶ 결과
                    (의도 명료화          (명확·구조화)
                     맥락·제약·형식 보강)
```

깎는 에이전트가 하는 일은 대략 다음과 같다:

| 단계 | 하는 일 | 예시 |
|------|---------|------|
| **의도 추출** | 사용자가 진짜 원하는 결과가 무엇인지 명시화 | "정리해줘" → "표 형식으로 핵심 5개를 우선순위 순 정리" |
| **맥락 보강** | 빠진 배경·전제를 채움 | 대상 독자, 사용 환경, 기존 자료 |
| **제약 명시** | 길이·톤·금지사항·포맷을 못박음 | "300자 이내, 존댓말, 코드블록 사용" |
| **모호성 제거** | 여러 해석 가능한 표현을 하나로 | "최근" → "최근 3개월" |
| **출력 형식 지정** | 받고 싶은 구조를 미리 규정 | JSON / 마크다운 표 / 단계별 리스트 |
| **(선택) 역할 부여** | 페르소나·관점 지정 | "20년차 증권 분석가 관점에서" |

> **메타프롬프팅(meta-prompting)**: 프롬프트를 만들기 위해 LLM을 쓰는 기법 전반을 부르는 말. "프롬프트 깎는 에이전트"는 그중에서도 *사용자 입력을 실시간으로 다듬는* 형태다.

---

## 2. 언제 쓰는가 / 언제 안 쓰는가

만능이 아니다. 끼워 넣을 가치가 있는 경우와 오히려 방해되는 경우가 갈린다.

### 쓰면 좋은 경우
- **같은 종류의 요청을 반복**할 때 (예: 매번 종목 분석, 매번 회의록 요약) — 좋은 프롬프트 틀을 한 번 만들어두면 계속 재사용
- 사용자 입력이 **짧고 거친 게 일상**일 때 (모바일 입력, 음성 메모, 한 줄 요청)
- 출력 **품질의 일관성**이 중요할 때 (사내 산출물, 고객 응대)
- 도메인 지식이 필요해 **매번 같은 맥락을 붙여야** 할 때 (회사 용어, 정책, 포맷 규칙)
- 비전문가가 전문 작업을 시킬 때 — 좋은 프롬프트 작성 부담을 에이전트가 흡수

### 안 쓰는 게 나은 경우
- **일회성·탐색적 대화** — 다듬는 단계가 오히려 답답함
- 사용자가 이미 **정교한 프롬프트를 직접 쓰는 숙련자**일 때 — 깎다가 의도를 왜곡할 위험
- **지연(latency)이 치명적**인 실시간 응답 — 단계가 하나 늘면 그만큼 느려짐
- 요청이 **너무 다양해서 깎는 규칙을 일반화하기 어려운** 경우

---

## 3. 어떻게 만드는가

만드는 방식은 크게 세 가지다. 가벼운 순서대로.

### 방식 A — 시스템 프롬프트 한 장 (가장 가벼움)

별도 코드 없이, 깎는 역할만 정의한 시스템 프롬프트를 모델에 얹는다. 깎는 에이전트의 핵심은 결국 **"깎는 규칙을 적은 프롬프트"** 그 자체다.

```text
너는 '프롬프트 리파이너'다. 사용자의 거친 요청을 받으면,
실행하지 말고 먼저 아래 기준으로 다듬은 프롬프트를 출력하라.

[다듬기 체크리스트]
1. 의도: 사용자가 원하는 최종 결과물을 한 문장으로 명시
2. 맥락: 빠진 배경·대상 독자·전제를 추론해 보강 (불확실하면 [가정] 표시)
3. 제약: 길이·톤·포맷·금지사항을 명시
4. 출력형식: 받고 싶은 구조를 규정 (표/리스트/JSON 등)
5. 모호성: 여러 해석이 가능한 표현을 하나로 확정

[출력 형식]
## 다듬은 프롬프트
<완성된 프롬프트>

## 내가 채운 가정
- <보강한 부분과 이유>
```

이 한 장만으로도 충분히 동작한다. Claude Code라면 이걸 **서브에이전트 정의(`.claude/agents/prompt-refiner.md`)**로 만들면, 필요할 때만 호출되고 메인 컨텍스트를 더럽히지 않는다.

```markdown
---
name: prompt-refiner
description: 거친 요청을 받아 잘 구조화된 프롬프트로 다듬는다. "프롬프트 다듬어줘", "이 요청 깎아줘" 시 호출.
tools: []
---

(위 시스템 프롬프트 본문)
```

### 방식 B — 2단계 파이프라인 (자동 연결)

깎은 결과를 사람이 검토하지 않고 **곧바로 본 작업 모델에 자동으로 넘기는** 구조. 코드(API)로 짤 때 전형적인 형태다.

```python
# 의사코드 (Anthropic SDK)
def run(user_input: str) -> str:
    # 1단계: 프롬프트 깎기
    refined = client.messages.create(
        model="claude-haiku-4-5-20251001",   # 깎기는 가벼운 모델로 충분
        system=REFINER_SYSTEM_PROMPT,
        messages=[{"role": "user", "content": user_input}],
    ).content[0].text

    # 2단계: 깎은 프롬프트로 본 작업
    result = client.messages.create(
        model="claude-opus-4-8",              # 본 작업은 강한 모델
        messages=[{"role": "user", "content": refined}],
    ).content[0].text
    return result
```

포인트:
- **깎기 단계는 작고 빠른 모델(Haiku)**, 본 작업은 강한 모델(Opus/Sonnet)로 분리 → 비용·지연 최소화
- 깎기 시스템 프롬프트는 **프롬프트 캐시**에 태우면 반복 호출 시 거의 공짜 (5분 TTL)

### 방식 C — 사람 검토 루프 (Human-in-the-loop)

깎은 프롬프트를 **사용자에게 한 번 보여주고 승인/수정받은 뒤** 실행. 품질이 중요하고 실수 비용이 큰 작업에 적합.

```
거친 요청 ──▶ 깎기 ──▶ [사용자에게 제시] ──▶ 승인/수정 ──▶ 본 작업
                          ▲                    │
                          └────── 반려 ─────────┘
```

- 장점: 의도 왜곡을 사람이 잡아냄, 점점 더 정확해짐
- 단점: 한 단계 더 느려짐, 자동화 이점 일부 상실

### 만들 때 공통 설계 포인트

| 포인트 | 권장 |
|--------|------|
| **깎기용 모델** | 작고 빠른 모델(Haiku 등) — 깎기는 추론보다 형식화 작업 |
| **시스템 프롬프트 캐싱** | 반복 호출이면 캐시 필수 (비용·속도) |
| **가정 표시** | 에이전트가 채운 추론은 `[가정]`으로 명시 → 왜곡 추적 가능 |
| **과도 보강 방지** | "없는 요구사항을 지어내지 말 것" 명시 — 환각 제약 |
| **실행 금지 경계** | 깎는 단계에선 절대 작업을 실행하지 않도록 못박기 |
| **few-shot 예시** | 좋은 깎기 예시 2~3개를 프롬프트에 넣으면 품질 급상승 |

## 4. 장점

| 장점 | 설명 |
|------|------|
| **입력 부담 감소** | 사용자는 거칠게 던져도 됨. "정리해줘" 한 마디면 충분 |
| **출력 일관성** | 같은 틀로 깎이니 결과 품질·포맷이 균일 |
| **도메인 맥락 자동 주입** | 회사 용어·정책·포맷을 깎기 단계가 매번 붙여줌 |
| **재사용성** | 한 번 잘 만든 깎기 규칙을 모든 요청에 적용 |
| **모델 비용 최적화** | 깎기는 싼 모델, 본 작업은 강한 모델로 분리 가능 |
| **암묵지의 명시화** | "좋은 프롬프트의 조건"을 규칙으로 박제 → 팀 전체가 공유 |
| **비전문가 접근성** | 프롬프트 작성법을 몰라도 좋은 결과를 받음 |

## 5. 단점·한계

| 단점 | 설명 | 완화책 |
|------|------|--------|
| **지연 증가** | 단계가 하나 늘어남 → 응답이 느려짐 | 깎기용 모델을 가볍게, 캐시 활용 |
| **비용 증가** | 호출이 2번 = 토큰 2배 (일부) | 깎기는 Haiku, 시스템 프롬프트 캐싱 |
| **의도 왜곡 위험** | 에이전트가 사용자 의도를 잘못 추론해 엉뚱하게 깎음 | `[가정]` 표시 + 사람 검토 루프 |
| **과잉 보강(환각)** | 없던 요구사항을 지어내 프롬프트를 부풀림 | "지어내지 말 것" 제약 + 보강 최소화 |
| **단순 요청엔 과함** | "1+1" 같은 걸 깎는 건 낭비 | 짧은 요청은 깎기 우회(bypass) 라우팅 |
| **숙련자엔 족쇄** | 정교하게 직접 쓰는 사람에겐 방해 | 깎기 on/off 토글 제공 |
| **디버깅 복잡도** | 결과가 이상할 때 깎기 탓인지 본 작업 탓인지 구분 어려움 | 깎은 프롬프트를 로그로 남기기 |
| **블랙박스화** | 사용자가 실제 실행된 프롬프트를 모름 | 깎은 결과를 노출(투명성) |

## 6. 안티패턴

| 안티패턴 | 문제 | 해결 |
|----------|------|------|
| 모든 요청을 무조건 깎기 | 단순 질문도 느려지고 비쌈 | 길이·복잡도 기준 라우팅 |
| 깎은 프롬프트를 사용자에게 안 보여줌 | 의도 왜곡을 못 잡고 신뢰 하락 | 최소한 로그로 남기고, 가능하면 노출 |
| 깎기·본 작업을 같은 비싼 모델로 | 비용 2배 | 깎기는 가벼운 모델로 분리 |
| 깎기 규칙에 "최대한 자세히 보강" | 환각·프롬프트 비대화 | "필요한 만큼만, 없으면 가정 표시" |
| 깎는 단계에서 작업까지 실행 | 검토 기회 상실·역할 혼선 | "깎기 단계는 실행 금지" 명시 |

---

## 7. 정리

- 프롬프트 깎는 에이전트 = **요청과 실행 사이에 "다듬는 단계"를 하나 끼우는 패턴**
- 핵심 가치는 **입력 부담↓ + 출력 일관성↑ + 도메인 맥락 자동 주입**
- 만드는 법은 가벼운 순서로 ① 시스템 프롬프트 한 장 → ② 2단계 자동 파이프라인 → ③ 사람 검토 루프
- 대가는 **지연·비용·의도 왜곡 위험** — 가벼운 모델·캐싱·`[가정]` 표시·투명한 노출로 완화
- **반복적이고 일관성이 중요한 작업엔 강력**, 일회성·탐색·숙련자 작업엔 오히려 방해

> **한 줄 원칙**: 같은 요청을 반복하고 있고, 매번 비슷한 맥락을 손으로 붙이고 있다면 — 그 손작업이 바로 깎는 에이전트로 박제할 후보다.
