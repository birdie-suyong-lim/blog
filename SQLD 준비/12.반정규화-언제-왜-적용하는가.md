## 1. 반정규화는 정규화를 "안 한 것"이 아니다

반정규화(De-normalization)를 "정규화를 게을리해서 생긴 결과"로 오해하기 쉬운데, 정반대다. **정규화를 다 마친 상태에서, 성능(주로 조회 성능)을 위해 의도적으로 중복을 다시 허용하는 작업**이다. 순서가 중요하다 — 정규화 → 문제점(조인 비용, 조회 성능) 파악 → 반정규화. 처음부터 정규화를 안 하는 것과는 전혀 다른 이야기다.

## 2. 왜 필요한가: 정규화의 대가

[정규화 글](8.정규화-1NF부터-BCNF까지.md)에서 다뤘듯, 정규화를 철저히 할수록 테이블 수가 늘어난다. 테이블이 늘면 조회할 때 조인을 더 많이 해야 하고, 조인이 많아질수록 쿼리는 느려진다. 특히 자주 조회되는 통계성 데이터(예: 게시글의 댓글 수, 주문의 총액)를 매번 실시간으로 집계하면 대량 트래픽에서 성능 문제가 생긴다. 반정규화는 이 트레이드오프에서 **일부 데이터 무결성을 포기하고 조회 성능을 얻는** 선택이다.

## 3. 반정규화의 세 가지 대상

시험에서는 반정규화를 적용하는 대상을 테이블/컬럼/관계 세 가지로 구분해서 묻는다.

| 대상 | 방법 | 예시 |
|---|---|---|
| 테이블 반정규화 | 테이블 병합, 테이블 분할(수직/수평), 이력 테이블 추가 | 자주 같이 조회되는 두 테이블을 하나로 합침 |
| 컬럼 반정규화 | 중복 컬럼 추가, 파생 컬럼 추가, 집계 컬럼 추가 | 주문 테이블에 `총주문금액` 컬럼을 미리 계산해 저장 |
| 관계 반정규화 | 중복 관계 추가 | 자주 조회되는 조합에 직접 참조 관계(외래키)를 추가해 중간 테이블 조인을 생략 |

가장 흔하게 시험에 나오는 건 **컬럼 반정규화 중 집계 컬럼(파생 컬럼)**이다. `주문상세`를 매번 SUM해서 총액을 구하는 대신, `주문` 테이블에 `총액` 컬럼을 미리 만들어 두는 방식이다.

```sql
-- 반정규화 전: 매 조회마다 집계
SELECT SUM(수량 * 단가) FROM 주문상세 WHERE 주문번호 = 1001;

-- 반정규화 후: 미리 계산된 값을 그냥 읽음
SELECT 총액 FROM 주문 WHERE 주문번호 = 1001;
```

## 4. 반정규화의 대가: 정합성 관리 비용

반정규화로 얻는 성능만큼, 잃는 것도 있다. 위 예시에서 `주문상세`가 바뀌면(상품 추가/취소/수량 변경) `주문.총액`도 같이 갱신해줘야 한다. 이 동기화를 놓치면 두 값이 어긋나는 **정합성 문제**가 생긴다. 그래서 실무에서는 트리거나 애플리케이션 로직으로 동기화를 강제하는 장치를 같이 만든다.

**반정규화는 공짜가 아니다** — 조회는 빨라지지만 삽입/수정/삭제 시 추가로 관리해야 할 코드와 검증 부담이 늘어난다는 게 시험에서 자주 묻는 트레이드오프다.

## 5. 반정규화를 적용하기 전에 먼저 검토할 것

시험에서 "반정규화를 검토하기 전에 먼저 해야 할 일"을 묻는 문제가 나온다. 곧바로 테이블 구조를 바꾸기 전에:

1. 뷰(View)를 사용해서 해결할 수 있는지 검토
2. 인덱스 추가로 해결할 수 있는지 검토
3. 애플리케이션 레벨에서 캐싱으로 해결할 수 있는지 검토
4. 그래도 안 되면 반정규화 적용

즉 반정규화는 **최후의 수단**에 가깝다. "성능이 느리면 무조건 반정규화부터 한다"는 선지는 오답이다.

## 6. 시험 함정 정리

- "반정규화는 정규화 과정을 생략하는 것이다" → 틀림. 정규화를 완료한 뒤 필요에 따라 되돌리는 것이다.
- "반정규화는 항상 데이터 무결성을 해친다" → 완전히 틀린 말은 아니지만, 트리거나 배치, 애플리케이션 로직으로 정합성을 관리하면서 사용하는 것이 정석이다. "관리 불가능"이 아니라 "관리 비용이 추가로 든다"가 정확한 표현.
- "반정규화 대상은 자주 조회되는 데이터에 한정된다" → 대체로 맞는 방향이지만, 정확히는 "조회 성능이 중요한데 정규화로 인한 조인 비용이 그 성능을 해치는 경우"에 검토 대상이 된다는 조건까지 함께 기억해야 한다.
