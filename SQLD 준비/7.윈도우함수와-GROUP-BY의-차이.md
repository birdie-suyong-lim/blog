## 1. 핵심 차이: 행이 줄어드는가

GROUP BY와 윈도우 함수(분석 함수, Window Function) 둘 다 "집계"라는 점에서 헷갈린다. 하지만 근본적인 차이가 있다.

- GROUP BY: 여러 행을 하나의 그룹으로 **묶어서 행 수를 줄인다.** 결과는 그룹 개수만큼의 행만 남는다.
- 윈도우 함수: 집계 계산은 하되 **원본 행을 그대로 유지**한다. 각 행 옆에 "계산된 값" 컬럼이 하나 더 붙는 것뿐이다.

```sql
-- GROUP BY: 부서 개수만큼 행이 나옴 (개별 직원 정보는 사라짐)
SELECT 부서, AVG(급여)
FROM 직원
GROUP BY 부서;

-- 윈도우 함수: 직원 수만큼 행이 그대로 나옴 (각 직원 옆에 부서평균이 붙음)
SELECT 이름, 부서, 급여,
       AVG(급여) OVER (PARTITION BY 부서) AS 부서평균
FROM 직원;
```

"개별 데이터는 유지하면서 그룹 통계도 같이 보고 싶다"는 요구가 나오면 GROUP BY로는 안 되고 윈도우 함수를 써야 한다는 신호다.

## 2. OVER절의 구성 요소

```sql
함수명(...) OVER (
    PARTITION BY 그룹기준컬럼    -- GROUP BY와 유사한 역할, 없으면 전체를 하나의 그룹으로
    ORDER BY 정렬기준컬럼         -- 그룹 내 순서 (순위 함수·누적 계산에 필요)
    ROWS/RANGE BETWEEN ...       -- 계산에 포함할 행 범위(윈도우 프레임), 생략 시 함수별 기본값
)
```

`PARTITION BY`가 GROUP BY의 역할을, `ORDER BY`가 그룹 내에서의 순서를 정한다. 이 둘을 헷갈려서 `PARTITION BY` 자리에 `GROUP BY`를 쓰는 문법 오류가 흔하다 — 윈도우 함수 안에서는 GROUP BY라는 키워드 자체를 쓰지 않는다.

## 3. RANK, DENSE_RANK, ROW_NUMBER의 차이

동점 처리 방식만 다르다. 이게 SQLD 단골 문제다.

```sql
SELECT 이름, 급여,
       RANK()       OVER (ORDER BY 급여 DESC) AS 순위,
       DENSE_RANK() OVER (ORDER BY 급여 DESC) AS 조밀순위,
       ROW_NUMBER() OVER (ORDER BY 급여 DESC) AS 일련번호
FROM 직원;
```

급여가 [500, 500, 400, 300]인 4명이 있다고 하면:

| 이름 | 급여 | RANK | DENSE_RANK | ROW_NUMBER |
|---|---|---|---|---|
| A | 500 | 1 | 1 | 1 |
| B | 500 | 1 | 1 | 2 |
| C | 400 | 3 | 2 | 3 |
| D | 300 | 4 | 3 | 4 |

- RANK: 동점이면 같은 순위, **다음 순위는 동점 인원수만큼 건너뛴다**(1, 1, 3, 4).
- DENSE_RANK: 동점이면 같은 순위, 다음 순위는 **건너뛰지 않고 바로 이어진다**(1, 1, 2, 3).
- ROW_NUMBER: 동점이어도 **무조건 서로 다른 번호**를 매긴다(1, 2, 3, 4). 동점자 중 누가 1번이 될지는 ORDER BY 기준이 완전히 정렬되지 않으면 결정적이지 않다.

"공동 1위가 2명이면 3위는 존재하는가?"라는 질문에 RANK는 "그렇다(3위)", DENSE_RANK는 "아니다(바로 2위 다음 인원)"로 답이 갈린다는 점을 예시와 함께 외우면 헷갈리지 않는다.

## 4. GROUP BY와 윈도우 함수를 같이 쓸 수 있는가

가능하다. GROUP BY로 먼저 집계한 결과에 대해 다시 윈도우 함수로 순위를 매기는 경우가 실무에서도 자주 나온다.

```sql
-- 부서별 평균급여를 구하고, 그 부서 평균급여끼리 순위를 매김
SELECT 부서, AVG(급여) AS 부서평균,
       RANK() OVER (ORDER BY AVG(급여) DESC) AS 부서순위
FROM 직원
GROUP BY 부서;
```

실행 순서상 GROUP BY(그리고 HAVING)가 먼저 처리되어 그룹이 확정된 뒤, 윈도우 함수는 SELECT 단계에서 그 위에 얹힌다. 그래서 GROUP BY 결과 행(부서별 1행)마다 윈도우 함수가 계산된다.

## 5. 시험 함정 정리

- "윈도우 함수는 WHERE절에서 조건으로 쓸 수 있다" → 불가능. 윈도우 함수는 SELECT 처리 단계에서 계산되므로 WHERE(그보다 앞선 단계)에서는 아직 값이 존재하지 않는다. 결과를 필터링하려면 서브쿼리나 CTE로 감싼 뒤 바깥에서 걸러야 한다.
- "PARTITION BY 없이 OVER()만 쓰면 에러다" → 아니다. PARTITION BY를 생략하면 전체 결과셋을 하나의 파티션으로 취급한다.
