## 1. 왜 헷갈리는가

키 종류가 5~6개나 되는데 다 "행을 구분하는 데 쓰인다"는 비슷한 설명이라 뭉뚱그려 외우게 된다. 이걸 구분하는 방법은 하나다 — **범위가 넓은 것부터 좁은 것 순서로 포함 관계를 그려보는 것.**

```
슈퍼키 ⊃ 후보키 ⊃ 기본키
              └ 대체키 (후보키 중 기본키로 뽑히지 않은 나머지)
```

## 2. 슈퍼키(Super Key): 유일성만 있으면 됨

행을 유일하게 구분할 수 있는 속성(또는 속성 조합)이면 전부 슈퍼키다. 불필요한 컬럼이 껴 있어도 상관없다.

```
직원(사번, 이름, 주민번호, 부서)
```

`사번`만으로도 유일하게 구분되지만, `사번+이름`, `사번+부서`, `사번+이름+부서` 전부 유일성 조건은 만족하므로 다 슈퍼키다. 슈퍼키는 "최소성"을 요구하지 않는다는 게 핵심이다.

## 3. 후보키(Candidate Key): 유일성 + 최소성

슈퍼키 중에서 **더 이상 컬럼을 빼면 유일성이 깨지는, 딱 필요한 만큼만 남긴 것**이 후보키다.

```
사번+이름 → 슈퍼키지만 후보키는 아님 (이름을 빼도 사번 하나로 유일성 유지되므로 최소가 아님)
사번      → 후보키 (더 뺄 게 없음)
주민번호   → 후보키 (사번과 별개로 이것도 유일성+최소성 만족)
```

한 테이블에 후보키가 여러 개 있을 수 있다(위 예시에서 `사번`과 `주민번호` 둘 다 후보키). 이 중 하나를 대표로 뽑는 게 다음 단계다.

## 4. 기본키(Primary Key) vs 대체키(Alternate Key)

여러 후보키 중 **실제로 테이블의 대표 식별자로 채택된 것**이 기본키, **채택되지 않고 남은 나머지 후보키들**이 대체키(보조키)다.

```
후보키: 사번, 주민번호
→ 사번을 기본키로 채택
→ 주민번키는 대체키(선정되지 않은 후보키)로 남음
```

시험에서 "대체키는 후보키가 아니다"라는 식의 선지가 오답으로 자주 나온다. **대체키는 후보키의 부분집합**이지 별개의 자격 조건을 가진 키가 아니다.

## 5. 외래키(Foreign Key): 다른 테이블을 참조

외래키는 지금까지의 키들과 성격이 다르다. 앞의 네 가지(슈퍼/후보/기본/대체키)는 전부 **자기 테이블 안에서 행을 구분**하기 위한 것이지만, 외래키는 **다른 테이블의 기본키(또는 후보키)를 참조**해서 두 테이블의 관계를 표현하는 용도다.

```
부서(부서번호, 부서명)          -- 부서번호가 기본키
직원(사번, 이름, 부서번호)       -- 이 부서번호는 직원 테이블 입장에서 외래키
```

외래키는 유일성을 요구하지 않는다는 점도 자주 나오는 함정이다. 위 예시에서 직원 테이블의 `부서번호`는 여러 직원이 같은 부서번호를 가질 수 있으므로(중복 허용) 유일하지 않지만, 여전히 외래키로 유효하다.

## 6. 표로 정리

| 키 종류 | 유일성 | 최소성 | 다른 테이블 참조 |
|---|---|---|---|
| 슈퍼키 | O | 필요 없음 | X |
| 후보키 | O | O | X |
| 기본키 | O | O | X (후보키 중 대표로 선정) |
| 대체키 | O | O | X (후보키 중 선정 안 된 나머지) |
| 외래키 | 필요 없음 | 해당 없음 | O |

## 7. 시험 함정 정리

- "슈퍼키는 최소성을 만족해야 한다" → 틀림. 최소성은 후보키부터 요구되는 조건이다.
- "외래키는 유일한 값이어야 한다" → 틀림. 외래키는 중복을 허용한다(1:N 관계에서 N쪽 테이블의 외래키는 당연히 중복된다).
- "외래키는 반드시 다른 테이블의 기본키를 참조해야 한다" → 정확히는 참조 대상 테이블에서 **유일성이 보장되는 컬럼(기본키 또는 후보키)**을 참조하면 된다. 기본키가 가장 흔하지만 유일한 조건은 아니다.
- "한 테이블에 후보키는 하나뿐이다" → 틀림. 후보키는 여러 개 존재할 수 있고, 그중 하나만 기본키로 선택된다.
