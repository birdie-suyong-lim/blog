## 1. 핵심 차이: 필터링 시점

WHERE와 HAVING을 둘 다 "조건절"로만 외우면 문제를 틀린다. 차이는 **언제 필터링이 일어나는가**다.

```
FROM → WHERE → GROUP BY → HAVING → SELECT → ORDER BY
```

- WHERE: 그룹으로 묶기 **전에**, 개별 행 단위로 걸러낸다.
- HAVING: 그룹으로 묶은 **후에**, 그룹(집계 결과) 단위로 걸러낸다.

이 실행 순서가 SQLD의 거의 모든 관련 문제의 근거다.

## 2. 왜 HAVING에서 집계함수를 쓸 수 있는가

WHERE 시점에는 아직 그룹핑이 일어나지 않았으므로 `SUM()`, `COUNT()` 같은 집계함수의 결과가 존재하지 않는다. 그래서 WHERE절에 집계함수를 쓰면 에러가 난다.

```sql
-- 에러: WHERE에는 집계함수 사용 불가
SELECT 부서, COUNT(*) 
FROM 직원
WHERE COUNT(*) > 5
GROUP BY 부서;

-- 올바른 방법: 그룹핑 이후 조건이므로 HAVING
SELECT 부서, COUNT(*) 
FROM 직원
GROUP BY 부서
HAVING COUNT(*) > 5;
```

## 3. 같은 조건도 WHERE와 HAVING에 넣으면 결과가 다르다

```sql
-- (A) WHERE로 개별 행을 먼저 거른 뒤 그룹핑
SELECT 부서, COUNT(*)
FROM 직원
WHERE 급여 >= 3000
GROUP BY 부서;

-- (B) 그룹핑 전체를 한 뒤 그룹 조건으로 거름
SELECT 부서, COUNT(*)
FROM 직원
GROUP BY 부서
HAVING SUM(급여) >= 3000;
```

(A)는 "급여 3000 이상인 직원만 센 인원수"고, (B)는 "부서 전체 급여 합이 3000 이상인 부서만" 남긴다. 조건에 들어간 컬럼이 같아 보여도 **필터링 대상이 행이냐 그룹이냐**가 완전히 다른 결과를 만든다. 시험에서 이 둘을 섞어 놓고 "결과가 같은가?"를 묻는다.

## 4. WHERE와 HAVING을 같이 쓸 수 있는가

가능하다. 실행 순서상 WHERE가 먼저 적용되어 그룹핑 대상 행 자체를 줄이고, 그 다음 그룹핑 결과에 HAVING이 적용된다.

```sql
SELECT 부서, AVG(급여)
FROM 직원
WHERE 입사년도 >= 2020        -- ① 행 필터: 2020년 이후 입사자만
GROUP BY 부서                  -- ② 부서별로 그룹핑
HAVING AVG(급여) >= 4000;      -- ③ 그룹 필터: 평균급여 4000 이상 부서만
```

성능 관점에서도 WHERE로 최대한 먼저 걸러내는 것이 유리하다. GROUP BY 대상 행이 줄어들기 때문이다.

## 5. 시험 함정 정리

- "HAVING은 GROUP BY 없이 못 쓴다" → 표준상 가능하지만(전체를 하나의 그룹으로 취급), SQLD 문제에서는 GROUP BY와 세트로 나오는 경우가 대부분이니 실행 순서를 기준으로 판단할 것.
- "WHERE 조건에 그룹핑 컬럼(GROUP BY에 쓰인 컬럼)은 못 쓴다" → 틀린 말. WHERE는 그룹핑 이전이므로 원본 컬럼이면 무엇이든 조건으로 쓸 수 있다. 못 쓰는 건 집계함수뿐이다.
