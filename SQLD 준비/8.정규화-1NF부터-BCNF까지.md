## 1. 정규화가 뭔지부터

정규화(Normalization)는 "데이터 중복을 없애고, 하나의 사실은 한 곳에만 저장되도록 테이블을 쪼개는 과정"이다. 목적을 먼저 기억해야 각 단계를 외우기 쉽다 — **모든 단계는 결국 "이상현상(Anomaly)을 없앤다"는 하나의 목표를 향해 간다.**

정규화가 어렵게 느껴지는 이유는 1NF, 2NF, 3NF, BCNF가 전부 "OO 종속을 제거한다"는 비슷한 말로 설명되기 때문이다. 각 단계가 **무엇을 제거하는지**로 구분하면 헷갈리지 않는다.

| 단계 | 무엇을 없애는가 |
|---|---|
| 1NF | 반복/다중값 속성 (한 칸에 여러 값) |
| 2NF | 부분 함수 종속 (기본키 일부에만 종속된 컬럼) |
| 3NF | 이행 함수 종속 (기본키가 아닌 컬럼에 종속된 컬럼) |
| BCNF | 후보키가 아닌 것이 결정자인 경우 |

## 2. 1NF: 한 칸에는 값 하나만

컬럼 하나에 여러 값이 들어있으면(반복 그룹, Repeating Group) 1NF를 위반한다.

```
위반: 주문번호 | 상품목록
      1001    | 사과, 바나나, 우유

해결: 주문번호 | 상품
      1001    | 사과
      1001    | 바나나
      1001    | 우유
```

"한 셀에 한 값만" — 이것만 지키면 1NF다. 가장 직관적이라 시험에서도 정답률이 높은 편이다.

## 3. 2NF: 부분 함수 종속 제거

2NF부터 "함수 종속(FD, Functional Dependency)" 개념이 필요하다. **기본키가 복합키(컬럼 2개 이상)일 때만 의미가 있는 단계**라는 게 핵심 — 기본키가 컬럼 하나뿐이면 자동으로 2NF를 만족한다.

```
주문상세(주문번호, 상품번호, 상품명, 주문수량)
기본키: (주문번호, 상품번호)
```

여기서 `상품명`은 `상품번호`만 알아도 정해진다. 즉 기본키 전체가 아니라 **일부(상품번호)에만 종속**되어 있다 — 이게 부분 함수 종속이다. `주문수량`은 주문번호+상품번호 조합 전체가 있어야 정해지므로 완전 함수 종속이고 문제 없다.

```
해결: 
주문상세(주문번호, 상품번호, 주문수량)   -- 완전 종속만 남김
상품(상품번호, 상품명)                    -- 부분 종속 컬럼을 분리
```

## 4. 3NF: 이행 함수 종속 제거

3NF는 **기본키가 아닌 컬럼끼리 서로 종속 관계**가 있을 때 문제가 된다. A → B, B → C면 A → C도 성립하는데(이행성), 이때 C가 B를 거쳐서 간접적으로 결정되는 것을 이행 함수 종속이라 한다.

```
직원(사번, 이름, 부서번호, 부서명)
기본키: 사번
```

`부서명`은 `사번`으로 결정되긴 하지만, 정확히는 `사번 → 부서번호 → 부서명` 순서로 결정된다. 기본키가 아닌 `부서번호`를 거쳐서 `부서명`이 정해지는 것 — 이게 이행 종속이다.

```
해결:
직원(사번, 이름, 부서번호)
부서(부서번호, 부서명)
```

## 5. 2NF와 3NF, 뭐가 다른지 헷갈릴 때

둘 다 "종속 관계를 끊고 테이블을 나눈다"는 결과는 비슷해서 구분이 잘 안 된다. **기준이 되는 지점**으로 구분하면 된다.

- 2NF: 기본키 "안에서" 문제(복합키의 일부 vs 전체)
- 3NF: 기본키 "밖에서" 문제(일반 컬럼끼리의 종속)

기본키가 단일 컬럼(예: 사번 하나)이면 2NF 위반은 애초에 생길 수 없고, 그 테이블에서 정규화 문제가 있다면 그건 항상 3NF(이행 종속) 문제다. 문제에서 기본키가 단일 컬럼인데 "2NF를 위반했다"는 선지가 나오면 그건 함정이다.

## 6. BCNF: 결정자가 후보키가 아닌 경우

3NF까지 만족해도 예외적으로 이상현상이 남는 경우가 있다. **결정자(다른 컬럼을 결정짓는 컬럼)가 후보키가 아닐 때** 발생한다.

```
수강(학생, 과목, 교수)
- 한 학생은 한 과목을 한 교수에게만 수강 (교수 한 명이 여러 과목을 담당 가능)
- 학생+과목 → 교수 (기본키)
- 교수 → 과목 (교수 한 명당 담당 과목은 하나로 고정이라고 가정)
```

이때 `교수`는 기본키의 일부도 아니면서 `과목`을 결정하는 결정자 역할을 한다. 즉 결정자(교수)가 후보키가 아니다 — 3NF까지는 통과하지만 BCNF는 위반한다.

BCNF는 SQLD보다는 SQLP(고급)에서 더 자주 나오지만, "3NF를 만족해도 이상현상이 남을 수 있다"는 사실 자체는 SQLD 지문에도 등장하므로 개념만은 알아두는 게 좋다.

## 7. 시험 함정 정리

- "정규화 단계가 높을수록 무조건 좋다" → 아니다. 정규화를 많이 할수록 테이블 수가 늘어 조인 비용이 커진다. 그래서 실무에서는 조회 성능을 위해 일부러 반정규화를 하기도 한다. ([반정규화 글](12.반정규화-언제-왜-적용하는가.md) 참고)
- "기본키가 하나의 컬럼이면 2NF는 신경 쓸 필요 없다" → 맞는 말이다. 부분 함수 종속은 복합키에서만 발생할 수 있다.
- "정규화는 성능을 높이기 위한 작업이다" → 정규화의 1차 목적은 데이터 무결성(중복 제거, 이상현상 방지)이지 성능이 아니다. 오히려 정규화가 심할수록 조회 성능은 떨어질 수 있다.
